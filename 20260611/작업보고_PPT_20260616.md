# 2. 운영로그 파서 V2

운영자 **메신저 채팅** → 구조화된 **장애 목록(Episode)** = 룰베이스 채점용 **정답지**.

```
"Deadlock 으로 OHT 정체중입니다"  →  Episode: 18:34 / 정체·병목 / OHT
```

| 단계 | 처리 |
|---|---|
| 카테고리 6종 | fault / action / recovery / rollback / cancel / normal |
| 장애유형 7종 | 정체·병목 / 브릿지 / 리프터 / CNV / MLUD / 통신·에러 / 기타 |
| **Episode 묶기** | 장애 1건에 여러 채팅 = 1건으로 (거품 제거) |
| 제외 처리 | orphan(선행 fault 없는 복구), CAPA변경(운영자 조치) |

> **효과**: 메신저 62건 메시지 → 실제 장애 **4건**(예시 5/25). 과대 카운트 제거 후 룰베이스 성능을 **정확히** 측정 가능.

# 3. 리프터 주변 HID 파악

R-C 룰(리프터 **역증가** 감지)이 단순 카운트를 넘어 **문제 리프터를 개별 ID로 특정**.

| 항목 | 내용 |
|---|---|
| 감지 신호 | M16HUB 리프터 **역증가 개수** (`rev_count`) + 20분 트렌드 |
| **개별 HID 파악** | `rev_lids` = 역증가 리프터 ID 목록 |
| 실측 예시 | `역증가 5개: 6ABL6021, 6ABL6031, 6ABL6032, 6ABL0112, 6ABL0121` |
| 임계 | 역증가 ≥ 4개 → R-C 발동 |

**활용**: "리프터 정체" 라는 막연한 알람이 아니라 **어느 리프터(HID) 가 문제인지** 즉시 지목 → 운영자가 해당 설비만 점검.

> 분석 도구 연동: 발동이벤트 한 줄 → 점수·룰·**원본 컬럼·실측값** 자동 분해 (`한줄분석` / 웹분석기 / LLM 스킬)
