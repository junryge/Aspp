## 전체 구조 — 두 개의 데이터가 만나서 평가된다

```
 ┌─────────────────────────────┐        ┌──────────────────────────────┐
 │  [A] 룰베이스 (예측)        │        │  [B] 운영로그파서v2 (정답지) │
 │  hubroom_predictor.py       │        │  운영로그_파서_v2.py         │
 │                             │        │                              │
 │  설비 센서 데이터 (매분)    │        │  운영자 메신저 채팅 로그     │
 │       ↓                     │        │       ↓                      │
 │  8개 영역 룰 평가 → 점수    │        │  채팅을 장애 단위로 구조화   │
 │       ↓                     │        │       ↓                      │
 │  사건단위.csv               │        │  episode.csv                 │
 │  (언제/어디서/무슨 증상)    │        │  (실제 발생한 장애 = 정답)   │
 └──────────────┬──────────────┘        └───────────────┬──────────────┘
                │                                        │
                └────────────┬───────────────────────────┘
                             ↓
                  [C] 매칭 + 평가 (사건_예측검증.py)
                  "룰베이스가 실제 장애를 몇 분 먼저 맞췄나?"
                  → TP / FP / Miss / 리드타임
```

- **[A] 룰베이스** = 설비 데이터로 장애를 **미리 예측**하는 쪽 (우리가 평가받는 대상)
- **[B] 운영로그파서v2** = 운영자가 실제 겪은 장애를 메신저에서 추출한 **정답지 (채점 기준)**
- **[C] 매칭** = A 의 예측이 B 의 실제 장애를 얼마나 먼저/정확히 맞췄나 채점

## 운영로그파서v2 란? — 왜 필요한가

룰베이스가 "잘 맞췄는지" 채점하려면 **"실제로 무슨 장애가 언제 났는지" 정답지** 가 있어야 합니다.
그 정답은 **운영자들이 메신저에 남긴 채팅** 에 있습니다. 하지만 채팅은 사람 말이라 그대로 못 씁니다.

**운영로그파서v2 = 운영자 메신저 채팅 → 구조화된 장애 목록(정답지) 으로 변환하는 도구**

### 실제 변환 예시 (2026-05-25 18:34)
```
[원본 메신저 채팅]
  "Deadlock 으로 인해 OHT 정체중입니다."   ← 운영자가 친 한 줄

        ↓ 운영로그파서v2 가 분석

[구조화 결과 = Episode]
  발생시각 : 2026-05-25 18:34
  카테고리 : fault (장애 발생)
  장애유형 : 정체/병목   ← "정체" 키워드 감지
  장비     : OHT         ← "OHT" 추출
  → 이 Episode 가 정답지에 1건 등록됨
```

### 8단계 처리 과정
| 단계 | 하는 일 | 5/25 예시 |
|---|---|---|
| 1. 메시지 분리 | 채팅 로그를 작성자/시각/본문 단위로 분리 | 62건 메시지 |
| 2. 전처리 | 멘션·URL·이미지 제거 | — |
| 3. 카테고리 6종 분류 | fault/action/recovery/rollback/cancel/normal | 18:34 → fault |
| 4. 장애유형 7종 분류 | 정체/브릿지/리프터/CNV/MLUD/통신/기타 | 18:34 → 정체/병목 |
| 5. 장비/라인 추출 | 정규식으로 장비명 인식 | 18:34 → OHT |
| 6. **Episode 묶기** | 같은 장비·30분 내 메시지 = 1개 장애 | (장애 발생→조치→복구를 1건으로) |
| 7. Episode 라벨링 | 시작/종료/지속/심각도 계산 | — |
| 8. 출력 | episode.csv (정답지) 생성 | 4건 (제외 후) |

### 왜 "메시지 1건 = 장애 1건" 이 아닌가 (Episode 묶기)
운영자는 장애 1건에 대해 여러 번 채팅합니다:
```
17:04 "M16A ZT Max Capa 50% 변경 부탁"      ← action
17:05 "변경했습니다"                          ← (같은 건)
17:37 "M14 Max Capa 원복 부탁"                ← rollback (같은 건)
17:41 "M14B 변경(원복) 완료"                  ← (같은 건)
```
→ 메시지는 4건이지만 **실제 작업은 1건**. 운영로그파서v2 가 이걸 1개 Episode 로 묶어 **거품(과대 카운트)을 제거** 합니다.
→ 5/25 메신저 62건 메시지 → 실제 장애 19건 → 평가대상 4건 (아래 §2 제외 룰 적용).

## 3. 룰베이스 "사건" 정의

발동이벤트.csv 의 매분 데이터에서 S3 (확정) 조건이 만족된 연속 구간을 사건으로 묶음.

### 3.1 S3 (확정) 조건
```
S3 = any_RA AND (any_RD OR any_SLA OR any_RC) AND (any_RB OR flow_severe)
```
**5/25 16:57 사건 예시** (실제 triggered_rules):
> `M16HUB:RA+RA_sus+RB+RB_fast+RC+RD; M14:RA_sus+RB+RB_fast+RC...`
> → M16HUB 가 RA(반송지연) + RD(저장정체) + RB(큐누적) 동시 발동 → S3 확정

### 3.2 사건 묶기 (진동 흡수)
```
INCIDENT_END_GAP_MIN = 60분
```
S3 가 잠깐 깨졌다 60분 안에 다시 만족되면 **같은 사건** (refire_count++).
**5/25 16:57 사건 예시**: 17:57~21:08, **3시간 11분 지속**. 이 사이 점수가 오르내려도 60분 안 끊김 → 1개 사건으로 유지 (refire 0회).

### 3.3 사건 기록 임계 + 5단계 등급
```
점수 < 100 → 사건단위.csv 기록 X (노이즈)
점수 ≥ 100 → 등급 부여:
  관심 100~119 / 주의 120~129 / 경계 130~159 / 위험 160~219 / 발동 220+
```
**5/25 예시**: 02:00 사건(점수80), 09:07 사건(점수95), 13:11 사건(점수85) 은 **점수 100 미만이라 기록 제외**. 100+ 인 5건만 사건으로 인정.

## 4. 매칭 규칙 — "예측했다 / 못했다" 판단

### 4.1 매칭 윈도우
```
사건 구간 = [predict_time - 30분, end_time + 60분]
```
- **PRE (30분)**: 예측 30분 전부터 매칭 허용
- **POST (60분)**: 사건 종료 후 60분까지 매칭 허용 (복구 지연 고려)

**5/25 18:34 정체/병목 매칭 예시**:
> 룰베이스 16:57 사건 → 구간 = [16:27, 22:08] (predict 16:57-30분 ~ end 21:08+60분)
> 메신저 18:34 가 [16:27 ~ 22:08] 안에 들어옴 → **매칭 성공 (TP)** ✅

### 4.2 매칭 조건
메신저 Episode 의 `start_time` (운영자 인지 시각) 이 사건 구간 안에 들어오면 **TP**.

### 4.3 매칭 한도
```
MAX_LEAD_MIN = 120분
```
룰베이스가 메신저보다 **120분 이상 일찍** 떴으면 **다른 사건으로 간주** (매칭 제외).
→ 5/25 18:34 사건의 리드는 +98분 → 120분 이내라 정상 매칭. 만약 +130분이었다면 "우연히 겹친 다른 사건" 으로 보고 매칭 안 함.

### 4.4 1:1 최근접 매칭 ★ 중요
- 한 사건은 한 Episode 만 매칭 가능
- 한 Episode 는 시간이 **가장 가까운** 미사용 사건과 매칭

**5/25 16:14 / 16:17 매칭 예시** (왜 16:17 이 Miss 인가):
> 메신저 16:14, 16:17 둘 다 룰베이스 15:46 사건 구간 [15:16 ~ 18:45] 안에 있음.
> → 먼저 처리되는 16:14 가 15:46 사건을 가져감 (리드 +28분, 더 가까움).
> → 남은 16:17 은 매칭할 사건이 없음 (15:46 은 이미 사용됨, 16:57 사건은 구간 시작 16:27 보다 16:17 이 앞섬).
> → **16:17 = Miss**.
> 이는 "한 사건이 여러 메신저를 흡수해 성능을 과장" 하는 것을 막기 위한 보수적 기준.

## 5. 리드타임 계산 — "몇 분 전에 감지했나"

```
리드타임 = 메신저 발생시각 - 룰베이스 예측시각
```
| 부호 | 의미 | 5/25 사례 |
|---|---|---|
| **+ 양수** | 룰베이스가 먼저 (사전 예측 ✅) | 18:34 - 16:57 = **+98분** |
| 0 또는 - 음수 | 동시 또는 늦음 | (5/25 엔 없음) |

**5/25 사전 예측 3건 모두 양수**:
- 16:14 기타 ← 15:46 예측 = **+28분**
- 18:34 정체/병목 ← 16:57 예측 = **+98분** ⭐
- 21:23 기타 ← 21:05 예측 = **+18분**
- → 5/25 평균 리드타임 = **+48분** (메신저 등록 평균 48분 전 감지)

## 6. TP / FP / Miss 정의

| 분류 | 정의 | 5/25 사례 |
|---|---|---|
| **TP (정탐)** | 룰베이스 사건 ↔ 매칭 성공 | 16:14, 18:34, 21:23 → 3건 |
| **FP (오탐)** | 룰베이스 사건 ↔ 매칭 실패 | 12:42(112), 20:33(114) → 2건 |
| **Miss (놓침)** | Episode ↔ 매칭 실패 | 16:17 기타 → 1건 |

### 6.1 정밀도 / 재현율
```
정밀도 = TP / (TP + FP) = 3 / (3 + 2) = 60%   ← 5/25
재현율 = TP / (TP + Miss) = 3 / (3 + 1) = 75%  ← 5/25
```

### 6.2 FP 해석 주의사항 ★
**FP 가 모두 "헛알람"이 아님:**
- 메신저는 운영자가 "채팅으로 보고한 장애" 만 기록
- 5/25 12:42(점수112), 20:33(점수114) FP 는 → 진짜 헛알람일 수도, **운영자가 메신저에 안 올린 조용한 정체일 수도** 있음
- → **실제 정밀도는 측정치(60%)보다 높을 가능성**
