좋은 질문이다. 명세서가 "메신저 파싱"만 다뤘는데, 이걸 V4 룰베이스/ML/스킬과 연결하면 가치가 폭증한다. 깐깐하게 점검해서 추가 후보 7개 골랐다.

1순위 — 무조건 추가해야 할 3개
A. 시스템(룰베이스) vs 운영자(메신저) 교차 매트릭스 ★★★
이게 가장 큰 가치다. Episode CSV 만들고 끝나면 그냥 통계. V4 룰베이스 결과와 합치면 자동 false positive/negative 라벨이 나온다.

시나리오	의미	활용
룰베이스 S3 + Episode 일치	True Positive	룰베이스 신뢰
룰베이스 S3 + Episode 없음	False Positive	임계 보정 후보
룰베이스 정상 + Episode 있음	Miss (놓침)	룰 추가 후보
운영자 메신저 13건 + 룰베이스 0건	작업 인폼 (가짜)	자동 구분
입증된 사례: 5/8 사건 시스템 12:41 vs 운영자 16:16 → 3시간 35분 사전감지 (이미 5월_30일_검증결과.md에 있음). v2 Episode 가 자동으로 이 매트릭스를 뽑게 만들면 매달 자동 검증 가능.

현재 messenger_to_incidents.py: 메시지 단위 카운트 → 라벨 부정확
v2 Episode 출력 → XGBoost 학습 데이터 직접 품질↑
추가 작업 거의 없음 (Episode CSV → JSON 변환만)
C. 작업 인폼 vs 실제 사건 자동 구분 ★★★
5/12 사례 (검증보고서 P.4): 메신저 13건 중 다 "AI Close 작업 사전 인폼". 룰베이스 S3 0% → 시스템과 교차하면 자동으로 "작업 인폼" 라벨링.

IF episode 키워드 = action(Close/Open) AND 룰베이스 score < 30:
    episode_type = "MAINTENANCE_NOT_FAULT"
    fault_count = 0  ← 라벨 B에서 제외
→ 라벨 B 정확도 직접 향상.

E. severity 자동 추정 (지속시간 기반)
명세서엔 키워드만 있지만 실데이터 키워드 단서 20건뿐. 지속시간으로 자동 추정:

duration < 15분  → LOW
duration 15~60분 → MEDIUM  
duration > 60분  → HIGH
영향 영역 ≥ 2개   → +1 등급
→ 운영자 키워드 의존 없이 객관적 severity.

F. 운영자 KPI 자동 (MTTD/MTTR)
Episode 한 번 만들면 다음 KPI 무료로 나온다:

MTTD (Mean Time To Detect): detect_time - start_time — 운영자 인지 지연
MTTR (Mean Time To Recover): end_time - start_time — 복구 소요 시간
장비별/팀별/시간대별 평균
→ 운영팀 성과 평가/개선 도구.

G. 반복/만성 장애 패턴 자동 감지
같은 장비가 7일 내 ≥ 3회 Episode → CHRONIC_DEVICE
같은 라인이 24시간 내 ≥ 5회       → HOTSPOT_LINE
같은 시간대 반복                    → SCHEDULED_ISSUE 후보
4ABLD131 사례: 5월에 8일 간격 2회 → 만성 X / 더 모으면 만성 패턴 자동 감지.

→ 정비/교체 우선순위 자동 제안.
