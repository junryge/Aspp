# AI 모델 학습 및 활용 수준 보고서

**작성일:** 2025년 12월  
**대상 시스템:** AMHS 장애 예측 모델 (V8)  
**작성 목적:** 현재 수준 평가 및 2026년 목표 수립

---

## 1. 현재 수준 (2025년)

| 평가 항목 | 점수 | 근거 |
|-----------|------|------|
| **모델 개발** | 3.0/5.0 | XGBoost 기반 수치형/범주형 2개 모델 구축, V8까지 반복 개선, 32개 feature 활용 |
| **학습 프로세스** | 2.0/5.0 | 수동 Python 스크립트 실행, GPU/CPU 자동 감지는 있으나 파이프라인 자동화 없음 |
| **모델 관리** | 1.5/5.0 | 파일명 버전 관리(V7→V8), pkl 파일 수동 저장, 체계적 레지스트리 없음 |
| **배포/서빙** | 1.5/5.0 | 로컬 평가 스크립트 수준, 실시간 서빙 시스템 없음 |
| **모니터링** | 1.0/5.0 | 평가 시 1회성 지표 확인, 지속적 성능 추적 없음 |
| **종합** | **1.8/5.0** | 모델 개발 역량은 있으나 MLOps 인프라 부재 |

### 현재 수준 상세

- **모델 구조:** XGBoost 기반, 수치형(MIN/MAX/AVG 3개 모델) + 범주형(3-Class) 구성
- **Feature:** V8 기준 32개 컬럼, 222개 파생 Feature 활용
- **성능:** 수치형 MAE < 10, R² > 0.65 / 범주형 정확도 > 70%
- **한계:** 수동 실행, 버전 관리 미흡, 실시간 예측 불가

## 2. 2026년 목표 수준

| 평가 항목 | 목표 점수 | 목표 상태 |

| **학습 프로세스** | 3.5/5.0 | 자동 재학습 파이프라인, 데이터 변화 감지 시 트리거 |
| **모델 관리** | 3.5/5.0 | 모델 레지스트리 도입, 버전/메타데이터 체계 관리 |
| **배포/서빙** | 3.0/5.0 | REST API 서빙, 스케줄 기반 배포 |
| **모니터링** | 3.0/5.0 | 대시보드 기반 성능 추적, 드리프트 감지 |
| **종합** | **3.4/5.0** | MLOps 기반 체계 구축 |

### 목표 달성 핵심 과제

1. **MLflow 도입:** 모델 등록, 추적, 배포 통합 관리
2. **자동화 파이프라인:** 데이터 수집 → 전처리 → 학습 → 평가 → 배포
3. **API 서빙:** 실시간 예측 서비스 구축
4. **모니터링 대시보드:** 성능 지표 실시간 추적

## 3. 모델 등록/추적/배포 스케줄 관리 시스템

### 추천 시스템: MLflow

| 기능 | 설명 |
|------|------|
| **Tracking** | 실험 파라미터, 메트릭, 아티팩트 자동 기록 |
| **Registry** | 모델 버전 관리, Stage(Staging/Production) 전환 |
| **Serving** | REST API 자동 생성, Docker 배포 지원 |
| **Scheduling** | Airflow/Prefect 연동으로 스케줄 관리 |

### 대안 시스템 비교

| 시스템명 | 특징 | 적합도 |
|----------|------|--------|
| **MLflow** | 오픈소스, 실험 추적/모델 레지스트리/배포 통합 | ⭐⭐⭐⭐⭐ 추천 |
| **Kubeflow** | K8s 기반, 대규모 파이프라인 | ⭐⭐⭐ (인프라 필요) |
| **DVC** | 데이터/모델 버전 관리 특화 | ⭐⭐⭐⭐ |
| **Weights & Biases** | 실험 추적/시각화 강점 | ⭐⭐⭐⭐ |
| **자체 개발** | 사내 요구사항 맞춤 | ⭐⭐⭐ (공수 큼) |
