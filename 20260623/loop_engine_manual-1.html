Loop Engineering Framework

AI한테 매번 시키지 말고,
AI를 시키는 루프를 짠다.

프롬프트를 짜던 자리에서 빠져나와 트랙(루프)을 설계한다. 반도체 FAB 폐쇄망에서 그대로 도는, 외부 의존성 0개의 파이썬 프레임워크.

핵심 개념

루프는 짓는 AI(generator) 와 검사하는 AI(verifier) 를 엮어 ‘조건 만족까지’ 자동 반복한다.

verifier 가 병목이다

generator(모델)는 이제 충분히 좋고 거의 공짜로 반복된다. 그 결과가 가치 있는지 판정하는 verifier 가 루프의 전부다. verifier 가 약하면 루프는 조용히 망하는 게 아니라 쓰레기를 자신있게 수백 번 생산한다. 그래서 “verifier 작성”이 새로운 프롬프트 엔지니어링이고, 당신의 도메인 지식이 곧 보상함수(reward function)다.

self-bias 경고

코드를 짠 모델은 자기 결과를 후하게 본다(실측: 자기채점 9.04 vs 독립채점 7.43). 그래서 generator 와 verifier 를 다른 모델로 강제 분리하고, 같은 계열이면 self_bias_warning 플래그를 자동으로 단다.

Open vs Closed 루프

  • Closed — 성공기준을 미리 못박고 매 스텝 평가 + 명시적 stop. 예측 가능·안전.
  • Open — 목표 + 느슨한 조건으로 넓게 탐색. 참신하지만 토큰 태우고 슬롭 위험.
  • 실전 트릭 — closed 하드체크를 ‘바닥’으로 깔고 그 위에 open 지시 하나 얹기. 기준 밑으로는 안 떨어지는 탐색.

설치 & 빠른 시작

pip 설치가 필요 없다. 압축을 풀고 import 하면 끝. 폐쇄망에는 zip을 base64로 옮긴 뒤 풀면 된다.

bash · 폐쇄망 반입
# 외부망에서
base64 loop_engine_pkg.zip > pkg.b64

# 폐쇄망에서
base64 -d pkg.b64 > loop_engine_pkg.zip
unzip loop_engine_pkg.zip
cd loop_engine_pkg

01

오토메이션 — 정해진 주기에 알아서 돈다

triggers.py · Scheduler · CronTrigger · IntervalTrigger · ManualTrigger

외부 cron이 없는 폐쇄망 prod를 위해 내장 스레드 스케줄러를 돈다. 표준 5필드 cron(분 시 일 월 요일, *·리스트·범위·스텝 + 일/요일 OR 시맨틱), N초 간격, 수동 발화 지원.

python
from loop_engine import Scheduler

sch = Scheduler()
sch.cron("morning", "0 9 * * 1-5", run_job)   # 평일 9시
sch.every("poll", 1800, run_job)            # 30분마다
sch.start()                                  # 백그라운드 스레드
02

워크트리 — 실행마다 격리, 충돌 없음

state.py · RunContext

git worktree 없이 runs/<run-id>/ 폴더로 실행을 분리한다. 동시에 여러 개 돌려도 서로 안 닿는다. 모든 산출물이 그 폴더에 박혀 추적된다.

python
from loop_engine import RunContext

run = RunContext("loop_workspace/runs", prefix="idc")
run.write_json("spec.json", {"goal": "..."})
run.set_status("shipped", rounds=2)
print(run.dir)   # .../runs/idc_20260623_..._a1b2c3
03

스킬 — 매번 처음부터 설명 안 하게

skills.py · Skill · SkillRegistry · SKILL.md

세션이 바뀌면 맥락이 백지가 되고, AI는 빈칸을 추측으로 메운다. 스킬은 프로젝트 지식을 SKILL.md(프론트매터 + 본문)에 박아두고, 루프가 매번 다시 읽게 한다.

python
from loop_engine import SkillRegistry

reg = SkillRegistry("loop_workspace/skills")   # */SKILL.md 자동 로드
print(reg.names())
ctx = reg.bundle(["logpresso_query"])        # 프롬프트 블록으로
04

커넥터 · 도구 — LLM과 ‘실제 일’에 손발

llm.py · LLMClient  |  tools.py · Tool · ToolRegistry · ToolAgent

OpenAI 호환 /chat/completions 를 urllib만으로 호출한다(목 transport 교체로 오프라인 테스트 가능). 도구는 파이썬 콜러블 — 내부 API, Logpresso 쿼리, 파일 작업. native function-calling 미지원 모델도 텍스트 JSON 프로토콜로 도구를 쓴다.

python
from loop_engine import LLMClient, LLMConfig, ToolRegistry

client = LLMClient(LLMConfig(endpoint="DEV", generator_model="qwen3.5-397b"))
print(client.complete("안녕"))

tools = ToolRegistry()
@tools.tool("query_lpql", "Logpresso 쿼리 실행", {"lpql": "쿼리문"})
def q(args): return run_internal_lpql(args["lpql"])
05

서브에이전트 — 짓는 AI ↔ 검사하는 AI

agents.py · Generator · VerifierAgent · same_family

둘을 반드시 다른 에이전트(가능하면 다른 모델)로 분리한다. VerifierAgent는 하드체크(CheckSuite) + LLM 루브릭(Rubric)으로 Verdict를 내고, 모델 계열이 같으면 self-bias 경고를 자동으로 단다.

python
from loop_engine import Generator, VerifierAgent, CheckSuite, Rubric, check_min_length

gen = Generator(client)                                   # 짓는 AI
ver = VerifierAgent(client,
    check_suite=CheckSuite([check_min_length(300)]),    # 하드 바닥
    rubric=Rubric("정확성·명료성 엄격 채점", pass_score=8.0),
    generator_model="qwen3.5-397b", verifier_model="glm-5")  # 다른 모델
06

상태 · 기억 — 에이전트는 잊어도 저장소는 안 잊는다

state.py · Artifact · Signal · WorkLog · Contract

Signal 은 루프를 복리로 만드는 단위(마찰·아이디어·기회 + 출처). WorkLog 는 세션 간 기억(작업 전 tail 읽고, 끝나면 append). Contract 는 루프당 계약(목표·워크플로·경계·백로그), 매 실행 전 읽힌다.

python
from loop_engine import WorkLog, ArtifactStore, Signal

log = WorkLog("loop_workspace/WORKLOG.md")
log.append("idc_loop", "shipped", "round 2, score 8.5")
print(log.tail(5))                # 최근 5개

루프 만들기

LoopSpec 으로 루프 한 벌을 정의하고 Loop(spec).run() 으로 돌린다. stop 조건은 (모든 critical 체크 통과 AND 루브릭 통과) 또는 max_rounds 도달. 미달이어도 최고점 라운드는 보존된다.

python · closed loop 전체
from loop_engine import (LLMClient, LLMConfig, Generator, VerifierAgent,
    CheckSuite, Rubric, LoopSpec, LoopMode, Loop,
    check_min_length, check_contains, check_no_placeholder)

client = LLMClient(LLMConfig(endpoint="DEV",
    generator_model="qwen3.5-397b", verifier_model="glm-5"))

spec = LoopSpec(
    name="amhs_brief",
    goal="AMHS 물류 혼잡 예측을 보고용 300자 이상으로 요약. 'AMHS' 포함, 미완성 표현 금지.",
    generator=Generator(client),
    verifier=VerifierAgent(client,
        check_suite=CheckSuite([check_min_length(300),
                                check_contains("AMHS"),
                                check_no_placeholder()]),
        rubric=Rubric("정확성·구조·실무성 엄격 채점", pass_score=8.0),
        generator_model="qwen3.5-397b", verifier_model="glm-5"),
    mode=LoopMode.CLOSED,
    max_rounds=4)

result = Loop(spec).run()
print(result.report())          # 라운드별 done/score
print(result.final_artifact)    # 최종 산출물
탐색을 얹고 싶을 때

mode=LoopMode.OPEN 과 open_instruction="제목은 더 도발적으로" 를 주면 하드체크 바닥은 유지한 채 탐색 지시가 얹힌다.

도구 쓰는 루프

LLM이 글만 쓰는 게 아니라 실제로 일을 하게 한다. ToolUsingGenerator 를 Loop 의 generator 자리에 끼우면 끝 — Loop/LoopSpec은 그대로다.

python
from loop_engine import ToolRegistry, ToolUsingGenerator

tools = ToolRegistry()
@tools.tool("query_idc", "FAB 구역 IDC 개수 조회", {"zone": "구역명"})
def query_idc(args):
    return run_internal_query(args["zone"])

# native=False(기본): 텍스트 JSON 프로토콜 → 모든 모델 호환
gen = ToolUsingGenerator(client, tools, model="gpt-oss-20b", max_steps=6)

# 단독 사용도 가능
from loop_engine import ToolAgent
out = ToolAgent(client, tools).run("M16A_3F IDC 보고서 작성")
print(out.final)            # 최종 산출물
print([s.tool for s in out.steps])  # 호출한 도구들

서버가 OpenAI tool-calling을 지원하면 native=True 로 네이티브 tool_calls 를 쓴다. 없는 도구 호출·프로토콜 위반은 자동 방어된다.

discover 단계 — 맥락 자동 주입

루프의 첫 동작은 discover: 무엇을 할지/어떤 맥락이 필요한지 모은다. DiscoverContext 가 스킬 + 공유 시그널 + 워크로그 + triage 를 한 덩어리로 묶어 context_provider 로 넘긴다.

python
from loop_engine import LoopSystem, TriageItem

system = LoopSystem()
spec = LoopSpec(..., context_provider=system.discover_context(
    skill_names=["logpresso_query"],     # 스킬 자동 주입
    include_open_signals=True,           # 공유 두뇌 시그널
    worklog_tail=8,                      # 세션 간 기억
    triage_provider=lambda: [TriageItem("긴급버그", 0)],  # 우선순위
).as_provider())

plan 단계 분리 (선택)

generator가 plan+execute를 동시에 하지만, 명시적 Planner를 끼울 수 있다. plan_once=True 면 첫 라운드만, False 면 매 라운드 재계획.

python
from loop_engine import Agent

planner = Agent("planner", "너는 계획가다", client, model="glm-5")
spec = LoopSpec(..., planner=planner, plan_once=True)

멀티루프 · 공유 두뇌(복리)

LoopSystem 이 작업공간·스킬·도구·시그널·워크로그·스케줄러를 한 번에 연결한다. 여러 루프가 같은 artifacts/signals/ 를 읽고 쓰면 서로의 발견을 이어받아 복리로 쌓인다.

python
from loop_engine import LoopSystem, LoopConfig, LLMConfig

system = LoopSystem(LoopConfig(llm=LLMConfig(endpoint="DEV")))
system.register(spec_a)
system.register(spec_b)

system.schedule_cron("loop_a", "0 9 * * 1-5")   # 평일 9시
system.schedule_every("loop_b", 1800)           # 30분마다
system.start()                                   # 백그라운드

# 한 루프의 발견 → 다른 루프가 픽업
system.emit_signal("이 키워드 전환되는데 콘텐츠 없음", kind="opportunity")

# 단발 실행(격리 run 디렉토리에서)
result = system.run_once("loop_a")

실전 연결 (DEMOS · Logpresso)

형 환경에 붙이는 세 갈래. 모두 Loop/LoopSpec은 그대로 두고 부품만 갈아끼운다.

  • DEMOS 117 에이전트 어댑터 — 기존 에이전트를 Generator/VerifierAgent 인터페이스로 래핑. SkillHarness의 Evaluate를 다른 모델 verifier로 강제 분리.
  • Logpresso 도구 등록 — query_carrier/query_oht 를 @tools.tool 로 감싸 루프가 직접 조회. (캐리어/필드 조건은 fulltext + ts_data_view_* 규칙 그대로)
  • BGE-M3 + FAISS RAG — 시방서/사내문서 검색을 discover_context(extra_provider=…) 로 discover 단계에 주입.
SkillHarness 적용 핵심

Draft→Test→Evaluate→Improve에서 Evaluate를 generator와 다른 모델로 분리하면, 9.04 vs 7.43 점수 갭이 스킬 품질에 그대로 실리는 걸 막는다. 이게 이 프레임워크가 강제하는 한 가지다.
