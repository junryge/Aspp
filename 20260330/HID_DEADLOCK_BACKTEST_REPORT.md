- **정체 지속성 룰(Rule 4)이 속도비 조건에 의존하므로 함께 비활성화되었다.**

이로 인해 경고 등급은 주로 포화도(Rule 1)와 유입/유출 불균형(Rule 3)의 조합으로만 발생하고 있으며, WARNING에 도달하려면 두 룰 모두 동시에 높은 점수를 받아야 하는 상황이다.

### 4.2 Rule 1 — HID 포화도 (발동률 90.6%)

**결과:** 주의 이상 등급의 90.6%에서 발동. 현재 시스템의 핵심 탐지 메커니즘이다.

**원인:**
- 포화율(HID_VALUE / VHL_COUNT_LIMIT) 분포에서 P75=0.324, P90=0.514, P95=0.622로, 일반적인 운영 상태에서도 0.5 이상이 빈번하게 발생한다.
- 현재 MEDIUM 임계값이 0.70인데, 이 구간이 전체의 약 4.1%(5,125건)이므로 적절한 민감도 수준이다.
- 다만 CRITICAL(0.95) 도달 건수가 265건(0.21%)으로, 최상위 등급 판정이 과도하게 희귀하다.

**포화율 분포 상세:**

| 구간 | 건수 | 비율 |
|---|---|---|
| 0 ~ 0.2 | 57,025 | 46.0% |
| 0.2 ~ 0.4 | 30,607 | 24.7% |
| 0.4 ~ 0.6 | 14,384 | 11.6% |
| 0.6 ~ 0.8 | 6,813 | 5.5% |
| 0.8 ~ 1.0 | 1,614 | 1.3% |
| 1.0 이상 | 21 | 0.02% |

**조치 제안:** CRITICAL 임계값을 0.95 → 0.80, HIGH를 0.85 → 0.65로 하향하여 탐지 민감도를 높일 것.

### 4.3 Rule 2 — 속도 저하 (발동률 0.2%)

**결과:** 사실상 비활성. 전체 12만 row에서 speed_ratio < 0.5인 건이 단 1건.

**원인:**
- FREE_FLOW_SPEED의 표준편차가 19.9이지만, HID별 평균 대비 비율(speed_ratio)로 환산하면 표준편차가 0.047로 극히 안정적이다.
- 이는 두 가지 가능성을 시사한다:
  1. **FREE_FLOW_SPEED가 실시간 주행 속도가 아닌 구간 설정 속도(기준 속도)일 가능성** — OHT가 정체 시 이 값이 변하지 않고, 정체는 HID_VALUE(체류 대수) 증가로만 반영될 수 있다.
  2. **OHT가 정체 시 아예 이동이 없어 HID_INOUT 테이블에 기록되지 않을 가능성** — 이동이 완료된 차량만 기록되므로, 정체 중인 차량의 속도 저하는 포착되지 않는다.

**speed_ratio 분포 상세:**

| 분위수 | 값 |
|---|---|
| P5 | 0.929 |
| P25 | 0.986 |
| P50 (중앙값) | 1.002 |
| P75 | 1.017 |
| P95 | 1.060 |
| 최소 | 0.452 |
| 최대 | 1.654 |

**조치 제안:** Rule 2를 속도비 기반에서 **유출 정지 기반**으로 재설계. `outflow_cnt = 0` (해당 HID에서 1분간 아무 차량도 출발하지 않음)을 정체 판단 대리 지표로 활용.

### 4.4 Rule 3 — 유입/유출 불균형 (발동률 67.2%)

**결과:** 주의 이상 등급의 67.2%에서 발동. Rule 1 다음으로 강력한 탐지 메커니즘이다.

**원인:**
- net_inflow(순유입 = 유입 - 유출) ≥ 3인 상황이 빈번하게 발생하며, 이는 HID 내 차량 축적이 활발하다는 의미이다.
- WARNING 이상에서 평균 net_inflow는 3.6으로, 분당 3~4대씩 축적되는 패턴이 주요 경고 원인이다.
- 특히 `outflow_cnt = 0`과 결합된 경우 (CRITICAL 4점) 2건의 IMMINENT 판정 모두에서 발동했다.

**조치 제안:** 현재 임계값(CRITICAL: net_inflow ≥ 5 AND outflow=0)은 적정. 다만 `net_inflow ≥ 3`의 MEDIUM 기준이 너무 관대할 수 있으므로, 데이터 축적 후 재검토.

### 4.5 Rule 4 — 정체 지속성 (발동률 0.1%)

**결과:** 사실상 비활성. 5건만 발동.

**원인:**
- Rule 4의 트리거 조건이 `saturation ≥ 0.85 AND speed_ratio < 0.5`의 **동시 충족**인데, speed_ratio < 0.5가 거의 발생하지 않으므로 Rule 4도 함께 무효화되었다.
- 이는 Rule 4 자체의 설계 결함이 아니라 Rule 2와 동일한 근본 원인(속도비 안정성)에 기인한다.

**조치 제안:** 속도비 조건을 제거하고, `saturation ≥ 0.70 AND net_inflow ≥ 2`가 N분 연속으로 변경. 포화도+유입 축적의 시간적 지속을 판단하도록 재설계.

### 4.6 Rule 5 — 연쇄 데드락 (발동률 31.1%)

**결과:** 주의 이상의 31.1%에서 발동. 공간적 확산 패턴 탐지에 유효하다.

**원인:**
- 인접 HID 그래프 분석 결과, 평균 인접 수 8.7, 최대 인접 20 (HID 110)으로, 충분한 공간 연결성이 확인되었다.
- 특히 HID 37~41 클러스터에서 Rule 5가 집중적으로 발동하였으며, 이는 5개 HID가 모두 서로 인접하여 하나의 포화가 인접 구역으로 전파되는 패턴이 관찰되었기 때문이다.

**인접 HID 그래프 통계:**

| 항목 | 값 |
|---|---|
| 전체 HID 쌍 | 3,653 |
| 빈번한 쌍 (≥10건) | 1,149 |
| 인접 맵 등록 HID | 181개 |
| 평균 인접 수 | 8.7 |
| 최대 인접 수 | 20 (HID 110) |
| 고립 HID | 2개 |

**조치 제안:** 현재 구조 유지. 다만 인접 HID 중 CRITICAL 2개 이상 조건의 adj_critical_count=2 기준은, 실제 클러스터 크기를 고려하면 적정하다.

1. **야간 배치 작업:** 야간에는 주간 대비 운영 인력이 적고, 대규모 배치(batch) 이동 작업이 집중된다. 특정 공정 완료 후 다량의 Lot이 동시에 이동을 시작하면, 특정 HID 구간에 유입이 급증한다.

2. **교대 시간 전후 효과:** 0시 전후의 야간 교대 전환 시점과, 6~7시 주간 교대 전환 시점에서 물류 패턴이 변동한다. 이 전환 구간에서 스케줄러가 밀린 작업을 한꺼번에 디스패치하는 경향이 있다.

3. **2~6시 피크 구간:** 이 시간대는 FAB 설비 가동률이 높으면서도 물류 관리 인력이 최소인 구간이다. 자동 디스패치 시스템이 부하 분산 없이 최단 경로로만 OHT를 배정할 경우, 특정 HID 클러스터(37~41)에 집중되는 패턴이 발생할 수 있다.

4. **18~23시 대비 저위험:** 18시는 데이터 시작점(피크 124,970건/시)이지만 WARNING은 3건에 불과하다. 이는 주간 시간대에는 운영 인력이 실시간 모니터링하면서 부하를 분산시키는 반면, 야간에는 자동 운영에 의존하기 때문으로 추정된다.

## 7. DEADLOCK_IMMINENT 상세 분석 (2건)

### 7.1 건별 상세

**Case 1: HID 146, 07:20**

| 항목 | 값 |
|---|---|
| 포화율 | **1.40** (한계의 140%) |
| 유입 | 5대 |
| 유출 | 0대 |
| 순유입 | +5 |
| 속도비 | 0.000 (유출 없어 측정 불가) |
| Rule 1 | 4점 (CRITICAL) |
| Rule 3 | 4점 (CRITICAL: 순유입≥5, 유출=0) |
| 총점 | 8점 → IMMINENT |

**원인:** HID 146은 VHL_COUNT_LIMIT 대비 140% 초과 상태에서 유출이 완전히 정지되었다. 이는 해당 구역에서 OHT가 진입은 하지만 빠져나가지 못하는 전형적인 데드락 전조 상태이다. 1분간 유입 5대가 모두 체류하면서 포화율이 급등했다.

**Case 2: HID 52, 07:22**

| 항목 | 값 |
|---|---|
| 포화율 | **1.00** (만석) |
| 유입 | 17대 |
| 유출 | 11대 |
| 순유입 | +6 |
| 속도비 | 0.999 (정상) |
| Rule 1 | 4점 (CRITICAL) |
| Rule 3 | 2점 (MEDIUM: 순유입≥3) |
| Rule 5 | 2점 (인접 HIGH) |
| 총점 | 8점 → IMMINENT |

**원인:** HID 52는 분당 17대 유입이라는 극단적 트래픽을 경험했다. 유출도 11대로 활발하지만, 유입이 유출을 6대 초과하면서 만석에 도달했다. Case 1과 달리 유출이 있으므로 완전한 데드락보다는 **일시적 과부하** 상태이다. 인접 HID도 HIGH 수준이어서 Rule 5까지 발동했다.

### 7.2 두 건의 공통점

- 발생 시각이 07:20, 07:22로 거의 동시간대 (주간 교대 직전)
- 포화율이 1.0 이상 (한계 도달 또는 초과)
- 07시는 야간 교대 종료 직전으로, 밀린 물류 작업 처리가 집중되는 시간대

## 9. 개선 방향

### 9.1 즉시 조치 (Week 1)

1. **Rule 2 재설계:** 속도비 → 유출 정지 기반
   - `outflow_cnt = 0` AND `inflow_cnt ≥ 2` → CRITICAL (4점)
   - `outflow_cnt ≤ 1` AND `inflow_cnt ≥ 3` → HIGH (3점)

2. **Rule 4 재설계:** 속도비 조건 제거
   - `saturation ≥ 0.70 AND net_inflow ≥ 2`가 3분 연속 → CRITICAL
   - 2분 연속 → HIGH

3. **포화율 임계값 하향:**
   - CRITICAL: 0.95 → 0.80
   - HIGH: 0.85 → 0.65
   - MEDIUM: 0.70 → 0.50

### 9.2 중기 조치 (Week 2~3)

4. **FREE_FLOW_SPEED 의미 확인:** 실시간 속도인지 구간 설정값인지 확인하여 Rule 2 방향 최종 결정
5. **HID 37~41 클러스터 FAB 레이아웃 확인:** 물리적 위치 파악 후 구간별 맞춤 임계값 설정
6. **야간 시간대 전용 임계값:** 0~7시 야간에는 임계값을 10~20% 하향하여 민감도 강화

### 9.3 장기 조치 (Week 4~)

7. **라벨 축적 후 ML 전환:** 수정된 룰베이스로 2~4주 라벨 축적 → LightGBM 베이스라인 모델
8. **HID 클러스터 자동 탐지:** 인접 그래프 기반 커뮤니티 디텍션 알고리즘 적용
9. **실시간 대시보드:** 룰 판정 결과를 실시간 모니터링할 수 있는 웹 대시보드 구축

## 10. 결론

14시간 야간 데이터(985,144건)에 대한 룰베이스 백테스트 결과, 5개 룰 중 3개(포화도, 유입유출, 연쇄)가 유효하게 작동하며, 2개(속도, 지속성)는 데이터 특성상 재설계가 필요하다.

가장 주목할 발견은 **HID 37~41 클러스터**로, 이 5개 인접 구역이 전체 경고의 62%를 차지하며 야간 0~7시에 집중 발생한다. 이는 FAB 내 특정 물류 병목 지점의 존재를 시사하며, 해당 구간의 물리적 위치 확인과 함께 OHT 디스패치 로직의 부하 분산 개선이 필요하다.

임계값 조정과 Rule 2/4 재설계를 통해 WARNING 비율을 현재 0.35%에서 목표 0.5~2.0%로 끌어올린 뒤, 2~4주간 라벨을 축적하여 ML 예측 모델로 전환하는 것을 권장한다.
