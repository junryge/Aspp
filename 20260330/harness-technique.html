SKILL EVALUATION FRAMEWORK

하네스(Harness) 기법

LLM 스킬의 품질을 체계적으로 측정하고 개선하기 위한
반복적 평가 프레임워크

01 — OVERVIEW

개요

하네스 기법의 핵심은 "Draft → Test → Evaluate → Improve" 반복 루프다. 정량적 벤치마크(assertions, pass rate, 토큰 사용량)와 정성적 인간 리뷰를 결합하여 스킬 품질을 체계적으로 끌어올린다.

핵심 철학: 오늘날의 LLM은 똑똑하다. 좋은 하네스가 주어지면 기계적 지시를 넘어 맥락을 이해하고 진짜 결과를 만들어낸다.
02 — CORE LOOP

핵심 루프

1

의도 파악 (Capture Intent)

사용자 인터뷰 → 스킬 목적, 트리거 조건, 출력 형식, 엣지 케이스 정리
2

스킬 작성 (Write SKILL.md)

YAML 프론트매터(name, description) + 본문 지시사항 + 번들 리소스(scripts, references, assets)
3

테스트 실행 (Spawn Runs)

with_skill + baseline을 동시에 스폰. 실행 중 assertions 초안 작성. timing.json에 토큰/시간 캡처.
4

평가/채점 (Grade & Review)

Grader Agent가 grading.json 생성 → 벤치마크 집계 → HTML 뷰어로 사용자 리뷰 → feedback.json 수집
5

스킬 개선 (Improve)

피드백 반영. 일반화 우선. WHY 설명. 반복 작업은 scripts/에 번들. 린 프롬프트 유지.
↻ 만족할 때까지 Step 3~5 반복 (보통 3~5 이터레이션)
03 — STRUCTURE

디렉터리 구조

skill-name/ ├── SKILL.md # 스킬 본체 ├── evals/ │ └── evals.json # 테스트 케이스 정의 ├── scripts/ # 번들 실행 스크립트 ├── references/ # 참조 문서 └── assets/ # 템플릿, 폰트, 아이콘 skill-name-workspace/ ├── iteration-1/ │ ├── eval-0-descriptive-name/ │ │ ├── with_skill/ │ │ │ ├── outputs/ # 스킬 적용 결과물 │ │ │ ├── grading.json # 채점 결과 │ │ │ └── timing.json # 실행 시간/토큰 │ │ └── without_skill/ │ │ └── outputs/ # 베이스라인 결과물 │ ├── benchmark.json │ └── benchmark.md ├── iteration-2/ … └── feedback.json # 사용자 리뷰
04 — SCHEMAS

평가 JSON 스키마

📋

evals.json

테스트 케이스 정의. prompt, expected_output, files, expectations(assertions)를 포함.

✅

grading.json

Grader Agent 출력. expectations[].text/passed/evidence + summary(pass_rate) + execution_metrics.

⏱

timing.json

실행 시간 캡처. total_tokens, duration_ms. 서브에이전트 완료 시 즉시 저장 (이후 복구 불가).

📊

benchmark.json

벤치마크 집계. with_skill vs without_skill: mean/stddev pass_rate, time, tokens + delta.

🔍

comparison.json

Blind A/B 비교. winner/loser + rubric(5점 척도) + output_quality 강점/약점 분석.

💡

analysis.json

Post-hoc 분석. 트랜스크립트 패턴 + instruction_following 점수 + improvement_suggestions.

evals.json 예시

JSON { "skill_name": "my-skill", "evals": [ { "id": 1, "prompt": "사용자가 입력할 실제 프롬프트", "expected_output": "기대 결과 설명", "files": ["evals/files/sample.pdf"], "expectations": [ "출력에 X가 포함됨", "Y 스크립트를 사용함" ] } ] }

benchmark.json 핵심 구조

JSON { "run_summary": { "with_skill": { "pass_rate": { "mean": 0.85, "stddev": 0.05 }, "time_seconds": { "mean": 45.0, "stddev": 12.0 }, "tokens": { "mean": 3800, "stddev": 400 } }, "without_skill": { "pass_rate": { "mean": 0.35, "stddev": 0.08 } }, "delta": { "pass_rate": "+0.50" } } }
05 — EXECUTION

실행 단계 상세

Step 1: Spawn

with_skill + baseline 동시 실행. 신규 스킬이면 no-skill 베이스라인, 개선 중이면 이전 버전 스냅샷을 베이스라인으로 사용.

Step 2: Assertions

실행 대기 중 assertion 작성. 파일 존재, 콘텐츠 검증, 구조 검증, 형식 검증. 주관적 품질은 정성 리뷰로.

Step 3: Timing

서브에이전트 완료 시 즉시 timing.json 저장. total_tokens, duration_ms — 이후 복구 불가.

Step 4: Grade

Grader → grading.json, aggregate_benchmark.py → benchmark.json, generate_review.py → HTML 뷰어 실행.

Step 5: Feedback — 빈 피드백은 만족을 의미. 구체적 불만 사항에만 집중하여 개선. feedback.json에 자동 저장됨.
06 — ADVANCED

고급 기법

Blind Comparison

독립 에이전트(Comparator)가 두 출력을 익명으로 비교. Rubric 5점 척도 + 강점/약점. Analyzer가 트랜스크립트 패턴 분석 + 구체적 개선 제안 도출.

Description Optimization

스킬 트리거 정확도 최적화. should_trigger / should_not_trigger 쿼리셋 → train 60% / test 40% → 최대 5회 반복 → test score 기준 best_description 선택.

07 — PRINCIPLES

개선 원칙

일반화
테스트 케이스에 오버핏하지 않는다. 소수 예시에서 작동하는 스킬은 의미 없다. 범용적으로 개선하고, 막히면 다른 메타포나 패턴을 시도.
린 프롬프트
효과 없는 지시를 제거한다. 트랜스크립트를 읽고 비생산적 패턴을 확인. 모델이 불필요한 작업에 시간을 낭비하면 해당 지시를 삭제.
WHY 설명
ALWAYS/NEVER 대신 이유를 설명. 모델은 맥락을 이해할 수 있다. 왜 중요한지를 전달하면 기계적 지시보다 더 효과적.
번들링
반복 작업을 scripts/에 번들한다. 모든 테스트 케이스가 동일한 헬퍼 스크립트를 만들면, 그건 스킬에 포함되어야 한다는 신호.
무놀라움
스킬 내용이 사용자를 놀라게 하면 안 된다. 악의적 코드, 익스플로잇, 설명과 다른 의도는 금지.
