## 2. 예측 가능 여부 판단

### 2.1 매크로 예측 (전체 흐름) → 가능

"10시에 어디가 바쁠까", "평균 사이클 타임이 얼마일까" 같은 전체 흐름 예측은 현재 데이터로 가능하다. FAB은 매일 비슷한 패턴으로 돌아가기 때문이다.

| 예측 대상 | 방법 | 정확도 |
|-----------|------|--------|
| 시간대별 트래픽 패턴 | 시계열 통계 | 높음 |
| 스테이션 처리량 예측 | 과거 패턴 기반 | 높음 |
| Load/Unload 시간 분포 | 통계적 분포 피팅 | 높음 |
| 가동률/적재율 추이 | 이동평균 | 높음 |
| OBS_BZ_STOP 발생 확률 | 시간대/구간 통계 | 중간 |

### 2.2 마이크로 예측 (개별 VHL) → 불가능

"5분 뒤 V00100이 어디 있을까"는 아무리 모델을 잘 만들어도 안 된다.

```
  불가능한 이유:
  - 1,033대가 서로 영향을 주고받음 (N-body 문제)
  - LineCost가 순환 의존 (경로→분포→비용→경로)
  - 작업 명령이 언제 올지 모름 (외부 입력)
  - layout.xml 없이 공간 관계 파악 불가

  → 날씨 예측과 비슷한 카오스 시스템
```

## 3. 룰 베이스 vs ML

### 결론: 룰 베이스가 정답

| 근거 | 설명 |
|------|------|
| 시스템 자체가 룰 베이스 | Dijkstra + LineCost + 페널티 가중치, 전부 사람이 정한 룰 |
| 룰이 코드에 다 있음 | `getVhlCountCost()` 공식, EMA 속도 갱신, 가중치(3000/5000/5000) |
| ML이 안 맞는 구조 | LineCost 순환 의존 → 입력-출력 관계가 고정되지 않음 |
| 룰을 아는 게 핵심 | 데이터로는 "왜 이 경로를 탔는지" 설명 불가, 룰이면 바로 설명 |

ML은 "패턴 이상 탐지" 정도의 부가적 용도에만 적합하다.

## 4. 10분/20분 예측은 가능한가?

### 4.1 룰 베이스로도 안 된다

Dijkstra는 **실시간 내비게이션**이다. "지금 어디가 막히니까 지금 이 길로 가라"이지, "10분 뒤에 어디가 막힐 거니까 미리 돌아가라"가 아니다.

### 4.2 핵심 한계: 작업 명령을 모른다

```
  룰 베이스 시뮬레이터가 맞추려면:

  알아야 하는 것         지금 있는 것
  ─────────────        ──────────────
  룰 (Dijkstra/Cost)    ✅ 코드에 있음
  layout (노드/엣지)     ❌ 없음
  현재 차량 상태          ✅ CSV로 있음
  향후 작업 명령          ❌ 모름  ← 핵심 한계
```

룰은 아는데 **입력값을 모르는 상황**이다. 계산기는 있는데 넣을 숫자를 모르는 것과 같다.

### 4.3 4가지가 다 모이면 가능

```
  룰(Dijkstra/LineCost)  ✅ 있음
  layout(노드/엣지)       ← 확보 필요
  현재 차량 상태           ✅ 있음
  작업 명령(CommandId)     ← 수집 필요
```

이 4개가 모이면 **시뮬레이션**(미래를 돌려보기)이 가능하다. 단, 예측이 아니라 시뮬레이션이다.

| 시간 | 정확도 | 이유 |
|------|--------|------|
| 10분 후 | 꽤 맞음 | 대부분의 차량이 현재 작업을 아직 수행 중 |
| 20분 후 | 흔들림 | LineCost가 계속 바뀌면서 실제와 차이 벌어짐 |
| 30분 이후 | 발산 | 작은 차이가 1,033대 상호작용으로 증폭 |

## 5. CommandId가 왜 중요한가

### 5.1 현재 상태

VHL_STATE_REPORT 메시지는 원래 **23개 필드**이다 (`OHT3/MAP_구축_가이드.MD` 기준).

```
  필드 0~20: 현재 수집됨 (21개 필드)
  필드 21: CommandId  ← 잘려서 안 들어옴
  필드 22: BayName    ← 잘려서 안 들어옴
```

### 5.2 CommandId가 있으면 할 수 있는 것

| 항목 | CommandId 없음 (현재) | CommandId 있음 |
|------|----------------------|---------------|
| 차량 작업 여부 | RunCycle/VhlCycle로 간접 추정 | 빈값=idle, 값 있음=작업 중 (직접 확인) |
| LineCost 계산 | idle/active 구분 부정확 | idle VHL × 3,000 vs 작업 VHL × 5,000 정확 구분 |
| 작업 발생 패턴 | 알 수 없음 | "이 시간대에 평균 N건/분 작업 발생" 파악 가능 |
| 10분/20분 예측 | 불가능 | 작업 패턴을 시뮬레이터에 넣어서 시뮬레이션 가능 |

### 5.3 결론

CommandId는 **"향후 작업 명령"이라는 마지막 퍼즐 조각**을 채우는 데이터이다.
수 주치 CommandId 데이터가 쌓이면 "이 시간대에 평균 몇 건의 작업이 발생하는가"를 알 수 있고, 그걸 시뮬레이터에 넣으면 10분/20분 시뮬레이션이 가능해진다.
