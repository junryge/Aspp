# OHT 경로 최적화 아키텍처

> Dijkstra 알고리즘 + 동적 LineCost 기반 VHL 경로 선택 시스템

---

## 1. 전체 시스템 구조 (Big Picture)

```
  ┌──────────────────────────────────────────────────────────────────┐
  │                        OHT 제어 시스템                            │
  │                                                                  │
  │   ┌──────────┐    ┌──────────────┐    ┌───────────────────┐     │
  │   │ 작업지시  │───>│  경로 계산기   │───>│   VHL에 경로 전달  │     │
  │   │ (Command)│    │ (Dijkstra)   │    │   (실제 주행)      │     │
  │   └──────────┘    └──────┬───────┘    └───────────────────┘     │
  │                          │                                       │
  │                    실시간 비용 조회                                │
  │                          │                                       │
  │                 ┌────────▼────────┐                              │
  │                 │   레일 네트워크   │                              │
  │                 │ (노드 + 엣지)    │                              │
  │                 │  동적 LineCost   │                              │

  └──────────────────────────────────────────────────────────────────┘
```

**한줄 요약**: 작업이 들어오면 → 레일 네트워크의 실시간 비용을 보고 → 가장 싼 길을 찾아 → VHL에게 알려줌

## 2. 레일 네트워크란?

FAB 천장의 OHT 레일을 **그래프(Graph)** 로 표현한 것입니다.

```
    [노드A]───엣지1───[노드B]───엣지2───[노드C]
       │                                    │
     엣지3                                엣지4
       │                                    │
    [노드D]───엣지5───[노드E]───엣지6───[노드F]
       
    노드 = 레일 위의 지점 (교차점, 분기점, 스테이션 등)
    엣지 = 노드와 노드를 잇는 레일 구간
```

실제로는 노드가 **수만 개**, 엣지도 **수만 개**인 거대한 네트워크입니다.

## 3. LineCost (엣지 비용) - 핵심 개념

### 3.1 "비용"이란?

> 이 레일 구간을 지나는 데 **얼마나 시간이 걸릴까?**를 숫자로 표현한 것

비용이 낮을수록 빠르게 통과 가능 = 좋은 길

### 3.2 비용 계산 공식

```
  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │   LineCost = 기본 통과시간                               │
  │            + (놀고 있는 차량 수 × 3,000)                  │
  │            + (일하는 차량 수   × 5,000)                   │
  │            + (대기 작업 수     × 5,000)                   │
  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

### 3.3 예시로 이해하기

두 경로 중 어디로 갈까?

```
  출발 ──[경로A]──> 도착       출발 ──[경로B]──> 도착

  경로A 엣지:                  경로B 엣지:
  ┌────────────────┐          ┌────────────────┐
  │ 길이: 100m      │          │ 길이: 150m      │
  │ 속도: 50m/min   │          │ 속도: 50m/min   │
  │ 기본시간: 2,000  │          │ 기본시간: 3,000  │
  │                 │          │                 │
  │ 놀고있는차: 2대   │          │ 놀고있는차: 0대   │
  │ 일하는차: 1대    │          │ 일하는차: 0대     │
  │ 대기작업: 1건    │          │ 대기작업: 0건     │
  │                 │          │                 │
  │ 페널티:          │          │ 페널티:          │
  │  2×3000=6,000   │          │  0              │
  │  1×5000=5,000   │          │  0              │
  │  1×5000=5,000   │          │  0              │
  │                 │          │                 │
  │ 총 비용: 18,000  │          │ 총 비용: 3,000   │
  │     (혼잡!)     │          │    (쾌적!)       │
  └────────────────┘          └────────────────┘
  
  결과: 경로B 선택! (비용 3,000 < 18,000)
  → 길은 더 길지만, 차가 없어서 훨씬 빠름
```

## 4. Dijkstra 알고리즘 - 최단 경로 찾기

### 4.1 쉬운 비유

> **네비게이션**과 같습니다.
> 실시간 교통정보(차량 수, 정체)를 반영해서 가장 빠른 길을 찾아주는 것

### 4.2 동작 순서

```
  ① 출발 노드에서 시작 (비용=0)
          │
          ▼
  ② 연결된 모든 엣지의 LineCost를 계산
          │
          ▼
  ③ 가장 비용이 낮은 노드를 선택
          │
          ▼
  ④ 도착지인가? ─── YES ──> ⑥ 경로 확정!
          │
          NO
          │
          ▼
  ⑤ 선택된 노드에서 다시 ②로 반복
          │
          ▼
  ⑥ 도착지까지의 경로를 역추적하여 반환
```

### 4.3 그림으로 보기

```
  출발(A)에서 도착(E)까지 최단 경로를 찾자!

        2,000        8,000
   (A) ────── (B) ────── (E)    경로1: A→B→E = 10,000
    │                    ▲
    │ 3,000        3,000 │
    │                    │
   (C) ────── (D) ──────┘      경로2: A→C→D→E = 9,000
        3,000
    
  Dijkstra가 찾은 답: A → C → D → E (비용 9,000)
  
  * B→E 구간이 혼잡(차량 많음)해서 비용이 8,000으로 높음
  * C→D→E는 돌아가지만 비용이 더 낮음
```

## 5. 왜 경로가 계속 바뀌는가?

### 5.1 LineCost가 실시간으로 변하기 때문

```
  ┌─── 시각 10:00:00 ───┐     ┌─── 시각 10:00:05 ───┐
  │                      │     │                      │
  │  엣지X:              │     │  엣지X:              │
  │  차량 1대 → 비용 낮음 │     │  차량 5대 → 비용 높음 │
  │                      │     │                      │
  │  → 경로A가 최적!     │     │  → 경로B가 최적!      │
  └──────────────────────┘     └──────────────────────┘

  불과 5초 만에 최적 경로가 바뀜!
```

### 5.2 비용을 바꾸는 4가지 요인

```
  ┌──────────────────────────────────────────────────┐
  │              LineCost 변동 요인                     │
  │                                                    │
  │   ┌──────────────┐     매 순간 VHL이              │
  │   │ ① 차량 수 변화 │     진입/이탈하면서            │
  │   │   (vhlIdMap)  │     엣지별 차량 수가 바뀜       │
  │   └──────────────┘                                 │
  │                                                    │
  │   ┌──────────────┐     작업 할당/완료 시            │
  │   │ ② 작업 상태   │     idle ↔ active 전환         │
  │   │  변화         │     → 페널티 가중치가 바뀜       │
  │   └──────────────┘                                 │
  │                                                    │
  │   ┌──────────────┐     새 명령 생성/완료 시         │
  │   │ ③ 대기 작업   │     스테이션 작업 수가 바뀜      │
  │   │  수 변화      │     → 페널티가 바뀜             │
  │   └──────────────┘                                 │
  │                                                    │
  │   ┌──────────────┐     VHL이 통과할 때마다          │
  │   │ ④ 속도 갱신   │     실제 속도를 이동평균으로     │
  │   │  (EMA방식)    │     갱신 → 기본시간이 바뀜      │
  │   └──────────────┘                                 │
  │                                                    │
  └──────────────────────────────────────────────────┘
```

## 6. 속도 갱신 방식 (EMA)

차량이 엣지를 지날 때마다 그 구간의 속도를 업데이트합니다.

```
  새 속도 = (기존 속도 × 0.6) + (방금 관측한 속도 × 0.4)
  
  예시:
  ┌───────────────────────────────────────────┐
  │ 기존 속도: 50 m/min                        │
  │ 방금 관측: 30 m/min (앞차 때문에 느렸음)    │
  │                                           │
  │ 새 속도 = 50 × 0.6 + 30 × 0.4            │
  │        = 30 + 12                          │
  │        = 42 m/min                         │
  │                                           │
  │ → 이 구간의 통과시간(비용)이 올라감          │
  │ → 다른 차량들은 이 구간을 피하게 됨          │
  └───────────────────────────────────────────┘
  
  가중치 0.6 : 0.4 = 급격한 변화 방지 (과거를 더 신뢰)
```

## 7. 페널티 가중치 정리

```
  ┌─────────────────────────────────────────────────────────┐
  │                    페널티 가중치                          │
  │                                                         │
  │   놀고 있는 차(Idle VHL)    1대당  +3,000 ms (3초)       │
  │   일하는 차(Work VHL)      1대당  +5,000 ms (5초)       │
  │   대기 작업(Work Dest)     1건당  +5,000 ms (5초)       │
  │                                                         │
  │   일하는 차 > 놀고 있는 차 : 작업중인 차량이              │
  │   Load/Unload로 멈출 가능성이 높아서 더 큰 페널티        │
  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

## 8. 전체 흐름 종합

```
  ┌─────────┐
  │ 작업 발생 │  "V00100, 스테이션A에서 스테이션B로 캐리어 운반해라"
  └────┬────┘
       │
       ▼
  ┌─────────────────────────────────┐
  │ 1. 출발/도착 노드 결정            │
  │    sourceNode = 스테이션A 노드    │
  │    destNode   = 스테이션B 노드    │
  └────────────┬────────────────────┘
               │
               ▼
  ┌─────────────────────────────────┐
  │ 2. Dijkstra 실행                 │
  │                                  │
  │    각 엣지마다:                   │
  │    ┌───────────────────────┐    │
  │    │ 기본시간 = 거리/속도    │    │
  │    │ + idle차 × 3,000      │    │
  │    │ + 작업차 × 5,000      │    │
  │    │ + 대기작업 × 5,000    │    │
  │    └───────────────────────┘    │
  │                                  │
  │    → 최소 비용 경로 선택          │
  └────────────┬────────────────────┘
               │
               ▼
  ┌─────────────────────────────────┐
  │ 3. 경로를 VHL에 전달              │
  │    [엣지1] → [엣지2] → [엣지3]  │
  └────────────┬────────────────────┘
               │
               ▼
  ┌─────────────────────────────────┐
  │ 4. VHL 주행 시작                  │
  │                                  │
  │    주행하면서:                    │
  │    - 각 엣지의 속도 EMA 갱신     │
  │    - 앞차 만나면 OBS_BZ_STOP     │
  │    - 분기점에서 경로 재계산 가능  │
  └────────────┬────────────────────┘
               │
               ▼
  ┌─────────────────────────────────┐
  │ 5. 상황 변화 시 경로 재계산       │
  │                                  │
  │    "아까는 경로A가 최적이었는데   │
  │     지금은 차가 몰려서            │
  │     경로B가 더 빠르네?"          │
  │                                  │
  │    → 새 경로로 변경!             │
  └─────────────────────────────────┘
```

## 9. XSOHS 데이터에서 보이는 증거

분석 리포트 데이터로 이 아키텍처를 확인할 수 있습니다:

| 현상 | 데이터 증거 | 아키텍처 연관 |
|------|------------|--------------|
| 속도 69.4%가 50 | 대부분 일반 주행 | 기본 속도 = 50 m/min |
| OBS_BZ_STOP 14.3% | 1,032대 중 거의 모든 차량 경험 | 앞차 감지 → 속도 EMA 하락 → 비용 증가 |
| 속도=0이 19.1% | 대기/정지 빈번 | 혼잡 구간 → LineCost 상승 → 우회 유도 |
| 가동률 82.3% | idle 17.7% | idle VHL 페널티(×3,000) 적용 대상 |
| 적재율 62.1% | 작업 차량 비율 | 작업 VHL 페널티(×5,000) 적용 대상 |
| 121,535개 OD쌍 | 같은 출발-도착이어도 다양한 경로 | 실시간 LineCost 변화로 매번 다른 경로 선택 |
