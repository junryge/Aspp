## 2. 데드락 (Deadlock)

### 2.1 정의

단방향 레일 위에서 차량끼리 서로 막혀서 **아무도 못 움직이는 상태**.

```
정상:
  ──→ V001 ──→ V002 ──→ V003 ──→
  (모두 같은 방향, 문제 없음)

데드락:
  ──→ V001 ──→ V002 ──→ V003 ──→ [차단] ←── V004
                                    ↑
                               여기서 막힘
  V003: 앞이 막혀서 정지
  V002: V003 때문에 정지
  V001: V002 때문에 정지
  → 연쇄 정지 → 아무도 못 움직임 → 데드락
```

### 2.2 발생 원인

| 원인 | 상황 |
|------|------|
| RAIL_CUT | 레일 구간 차단 → 앞차 정지 → 뒷차 연쇄 정지 |
| 분기점 충돌 | 두 방향 차량이 합류점에서 교착 |
| 적재/하차 정체 | 스테이션 작업 대기 → 뒤로 줄줄이 막힘 |
| OBS 연쇄 | 앞차 OBS(장애물정지) → 뒷차도 OBS → 연쇄 확산 |

### 2.3 데이터에서 보이는 징후

데드락이 오면 **OBS(장애물정지)가 급증**한다.

| 수준 | OBS 대수 | OBS 비율 | 상태 |
|------|---------|---------|------|
| 정상 | 120~160대 | 11~15% | 현재 이 범위 |
| 위험 | 180~200대 | 17~19% | 15:29에 198대까지 상승 후 자동 해소 |
| 데드락 | 300대 이상 | 30%+ | **2일간 미발생** |

### 2.4 2일간 실적

- 실제 데드락: **0건**
- 최대 OBS: **198대 (18.6%)** — 15:29 기록, 이후 자동 해소
- 해소 메커니즘: Dijkstra 알고리즘이 우회 경로를 재계산하여 해소

### 2.5 10분 후 데드락 예측 방법

```
현재 OBS 추이 관찰 (분 단위)

  OBS 120대 → 140대 → 160대 → 180대  (상승 추세)
  ────────────────────────────────────
  │         36분간 빌드업              │
  │    이 패턴 감지 시 10분 전 경고     │
  └────────────────────────────────────┘

판단 기준:
  - OBS < 160대, 변동 ±10대 이내  → 정상 (경고 불필요)
  - OBS 160~200대, 상승 추세       → 주의 (10분 내 악화 가능)
  - OBS > 200대, 계속 상승          → 위험 (데드락 임박)
```

## 3. 혼잡 구간 예측

### 3.1 HID_INOUT 기반 구간 속도

HID_INOUT 데이터에서 182개 HID 구간의 속도를 측정. 저속 구간은 **물리적 구조에 의해 고정적**이므로 10분 후에도 동일.

| 구간 유형 | 속도 특성 | 10분 후 예측 |
|-----------|----------|-------------|
| 고속 직선 구간 | 속도 90+ | 그대로 유지 |
| 분기/합류 구간 | 속도 60~80 | 그대로 유지 (구조적) |
| 스테이션 앞 | 속도 40~60 | 그대로 유지 (정차/감속) |
| 저속 병목 구간 | 속도 < 40 | 그대로 유지 (고정적) |

> 혼잡 구간은 레일 구조에 의해 결정되므로 **시간대와 무관하게 일정**.
> 큐가 1,200 → 1,600으로 올라가도 HID 속도 차이는 -0.4에 불과.

### 3.2 예외: RAIL_CUT 발생 시

RAIL_CUT(레일 차단)이 발생하면 해당 구간과 인접 구간의 혼잡도가 급변.

| 항목 | 내용 |
|------|------|
| 2일간 RAIL_CUT | 19건 (평균 1건/2.4시간) |
| 영향 | 차단 구간 ± 인접 HID 구간 속도 저하 |
| 감지 | RAIL_CUT 이벤트 발생 즉시 → 10분 후 영향 구간 예측 가능 |
