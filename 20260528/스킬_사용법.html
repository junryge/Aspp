2. 일반 스킬 vs 카파시 스킬 — 무엇이 다른가

두 스킬은 완전 독립이다. 같은 룰베이스를 다른 관점에서 정리한다.

구분 일반 스킬 카파시 스타일 스킬
목적 매분 사건 1건을 받아 빠르게 분류·대응 도메인 이해. 같은 문제를 보면 스스로 재구성할 수 있게
스타일 표·리스트·SOP 절차 15 강의 ("Lecture 0~15"). 직설적·왜·예시
주 사용자 운영 LLM 에이전트 (실시간 응답) 학습 단계 LLM (fine-tuning, few-shot 예시)
토큰량 ~2.5K (system 주입 가능) ~3.5K (학습용 또는 단발성 컨텍스트)
같이 쓸 수 있나? ✅ 둘 다 system prompt 에 넣어도 OK. 컨텍스트 윈도우만 충분하면.
언제 어느 것?
• 실시간 운영 (분류만 빠르게) → 일반 스킬 단독
• 모델 학습 / 도메인 이해 → 카파시 스킬 단독
• 사건 분석 / 사후 보고서 → 둘 다

3. LLM 별 사용 시나리오

Claude (API / Claude Code)

messages = [{
  "role": "system",
  "content": open("일반_에이전트_스킬.md").read()
}, {
  "role": "user",
  "content": event_csv_row
}]

팁 Claude 는 한국어 MD 그대로 잘 받음. 카파시 스킬은 prompt cache 사용 권장.

ChatGPT (OpenAI API)

system_prompt = open("일반_에이전트_스킬.md").read()

client.chat.completions.create(
  model="gpt-4o",
  messages=[
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": event_row}
  ]
)

팁 일반 스킬은 토큰 효율적. 카파시 스킬은 별도 fine-tuning 데이터로.

Gemini (Vertex AI)

system_instruction = open(
  "일반_에이전트_스킬.md"
).read()

model = GenerativeModel(
  "gemini-2.0-flash",
  system_instruction=system_instruction
)

팁 system_instruction 에 일반 스킬 주입. 컨텍스트 1M 이라 둘 다 OK.

로컬 LLM (Ollama / vLLM)

ollama run llama3.1 -s "$(cat 일반_에이전트_스킬.md)"

# 또는 카파시 스킬로 fine-tune:
# pip install axolotl 등으로 학습 데이터화

팁 소형 모델은 일반 스킬만. 14B+ 이면 둘 다 처리 가능.

4. 사용 워크플로우

1
룰베이스 코드 변경 — hubroom_predictor.py 의 임계 / 룰 수정.
2
룰 자동 추출 — python rules_extractor.py 실행.
→ rules_dump.json 갱신.
3
MD 동기화 검증 — 
python rules_extractor.py --check 일반_에이전트_스킬.md
python rules_extractor.py --check 카파시_스타일_에이전트_스킬.md
✅ 누락 없음 또는 ❌ 누락 N개 출력.
4
MD 수정 — 누락된 키워드 / 임계를 해당 MD 에 추가.
5
LLM 배포 — 두 MD 중 하나(또는 둘)를 system prompt 로 주입.
6
운영 검증 — 실제 사건 1건을 LLM 에 넣어 분류 결과 확인.

5. rules_extractor.py — 누락 검증 스크립트

5.1 동작

  1. hubroom_predictor.py 를 ast 로 파싱
  2. 32개 상수 / 9개 시그널 가중치 / 6개 등급 / 3개 Stage 조건 추출
  3. 모든 영역 / 룰 / 임계를 "필수 키워드 41개" 로 정규화
  4. MD 파일 텍스트에 41개 모두 등장하는지 점검
  5. 누락 시 STDERR + exit 1 (CI 통합 가능)

5.2 명령어

# 룰 추출만 (rules_dump.json 생성)
python rules_extractor.py

# MD 누락 검증
python rules_extractor.py --check 일반_에이전트_스킬.md
python rules_extractor.py --check 카파시_스타일_에이전트_스킬.md

# 출력 파일 이름 변경
python rules_extractor.py --out my_dump.json

5.3 출력 예시

✅ 룰 덤프 → rules_dump.json
   상수 32개
   영역 점수 시그널 9개
   위험도 등급 6개
   Stage 조건 3개

📋 누락 검증: 일반_에이전트_스킬.md
   필수 키워드 41개
   포함 41개
   ✅ 누락 없음

5.4 의존성

요구버전
Python≥ 3.8
표준 라이브러리argparse / ast / json / re / sys / pathlib
외부 패키지없음 ✅
CI 통합 가능. exit code 1 = 누락. Pre-commit hook 또는 GitHub Action 에서 python rules_extractor.py --check 일반_에이전트_스킬.md 실행하면 문서 자동 동기화 강제.

7. FAQ

Q. 일반 / 카파시 둘 다 system prompt 에 넣어도 되나? 가능. 합쳐서 약 6K 토큰. Claude / GPT-4o / Gemini 모두 충분한 컨텍스트. 단, 카파시는 "왜" 중심이라 운영 시에는 일반만으로도 충분.
Q. 룰베이스 변경하면 어떻게 동기화? 1. hubroom_predictor.py 수정
2. python rules_extractor.py 로 추출
3. --check 로 두 MD 검증
4. 누락된 키워드 / 임계를 MD 에 수동 추가
5. 다시 --check 통과까지 반복
Q. 다른 LLM (예: Mistral, Llama) 에서도 작동? 일반 스킬은 7B 이상 모델이면 분류 가능. 카파시 스킬은 14B+ 권장. 구조화된 표/임계가 많아 instruction-tuned 모델이 유리.
Q. RAG (검색 증강) 으로 쓸 수 있나? 가능. 두 MD 를 chunk 로 쪼개 vector store 에 저장 → 쿼리 시 관련 섹션만 retrieval. 일반 스킬은 섹션별 (R-A, R-B...) 분리가 깔끔.
Q. rules_extractor.py 가 ML 까지 검증? 아니다. 룰베이스 (hubroom_predictor.py) 만 검증. ML 부분 (train_xgboost.py, feature_builder_v41.py) 은 본 스크립트 범위 밖. ML 코드 변경은 별도 관리.
Q. fine-tuning 데이터로 쓰려면? 카파시 스킬 + 실제 사건 분류 사례를 (instruction, response) 쌍으로 변환: 
{"instruction": "다음 발동이벤트를 분류하라: [csv row]",
 "response": "유형 1 (R-A), 즉시 대응, M16HUB 9.0분 초과"}
약 50~100 쌍이면 7B 모델 LoRA 충분.
Q. 한국어가 아닌 영어 버전도 가능? LLM 에 "다음 문서를 영어로 번역하고 system prompt 로 사용하라" 라고 하면 됨. 단, 컬럼명 (예: AVGTOTALTIME1MIN) 은 원문 유지 필수.
Q. 누락 검증 키워드 41개는 어떻게 정해지나? rules_extractor.py 의 collect_required_terms 함수: 8 영역 + RA/RB 임계 영역 + 10 FLOW_NODES + 6 등급 + 9 시그널 + 핵심 키워드 (R-A, R-B, R-C, R-D, SLA, Sorter, MLUD, MAXCAPA, 4ABLD). 중복 제거 → 41개.
