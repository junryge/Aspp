1우리가 무엇을 하고 있나?

이천에 있는 SK하이닉스 반도체 공장 (M14, M16 빌딩) 에는 FOUP 이라는 통이 있습니다. 이 통 안에 웨이퍼 (반도체 칩의 원재료) 가 들어있고, 공정 진행에 따라 장비에서 장비로 이동해야 합니다.

FOUP 은 사람이 들고 다니지 않고, 천장에 설치된 레일 위를 OHT (무인 운반차) 가 들고 다닙니다. 마치 공항의 무인 셔틀이 정해진 경로로 다니는 것과 비슷합니다.

M14 빌딩과 M16 빌딩 사이의 모든 FOUP 은 M16HUBROOM (HUB) 이라는 중앙 공간을 통과합니다. 즉:

[M14 빌딩] ──→ [M16HUBROOM] ──→ [M16 빌딩] ↑↓ 허브가 막히면 전체 다 막힘
HUBROOM 이 정체되면 공장 전체 생산이 늦어집니다. 그래서 매분 매분 HUBROOM 상태를 자동으로 감시하고 점수를 매기는 것이 룰베이스의 목적입니다.

2꼭 알아야 할 용어 7개

FOUP — 웨이퍼가 담긴 통. 자판기 음료 한 캔 정도 크기. 한 번에 25장 정도 들어있음.
OHT — Overhead Transport. 천장 레일 위 무인 운반차. FOUP 을 들고 장비 사이를 이동.
FAB — Fabrication. 반도체 공장의 생산 라인. M14 FAB / M16 FAB 처럼 빌딩별로 부름.
M16HUBROOM (HUB) — M14 빌딩과 M16 빌딩의 FOUP 이 만나는 중앙 환승역. 3층 (3F) 에 위치.
LFT (리프터) — 층간 이동 엘리베이터. FOUP 을 다른 층으로 옮김. ZT 라고도 부름.
CNV (컨베이어) — 같은 층 내 이동 장치. FOUP 을 컨베이어 벨트로 다음 위치로 보냄.
큐 (Queue) — 처리 대기 중인 FOUP 줄. 큐가 길면 정체. 마트 계산대 줄을 생각하세요.

3왜 점수를 매기나?

운영팀 사람이 100개가 넘는 컬럼을 매분 직접 보고 "지금 정체인가?" 판단하기는 불가능합니다. 그래서 컴퓨터가 자동으로:

  1. 매분 데이터 (수백 개 컬럼) 를 읽어서
  2. "정체 신호" 라고 정의된 4가지 룰 (R-A, R-B, R-C, R-D) 을 검사
  3. 발동한 룰에 따라 점수를 가산
  4. 총 점수에 따라 위험도 등급 매김 (정상 → 관심 → 경계 → 주의 → 위험 → 매우위험)
  5. 3가지 룰이 동시에 발동하면 Stage 3 (확정) 사건으로 기록
병원의 응급실 트리아지와 비슷합니다. 환자가 들어오면 의사가:
• 체온 (R-A 같은 시간 지표)
• 혈압 (R-B 같은 누적 지표)
• 의식 상태 (R-C 같은 패턴 지표)
• 산소포화도 (R-D 같은 용량 지표)
를 모두 체크해서 "초록(정상) → 노랑(주의) → 빨강(위급)" 등급을 매기는 것과 같은 원리입니다.

4점수가 만들어지는 큰 흐름

전체 점수는 3 단계로 만들어집니다:

[원본 데이터] → [1단계: 룰 트리거 판정] → [2단계: 영역 점수] → [3단계: 통합 점수] 265 컬럼 True/False 0~50점 0~500점 (이 룰이 발동했나?) (한 영역의 점수) (전체 시스템 점수)

1단계 — 룰이 발동했나?

R-A, R-B, R-C, R-D 각 룰의 조건을 검사. 예: "지난 10분 동안 반송시간이 9분 넘은 적 있나?" → 예/아니오.

2단계 — 영역 점수 (0~50점)

한 영역 (예: M16HUB) 에서 발동된 룰 점수를 합산.
R-A 발동 → +10점, R-B 발동 → +10점, ... 모두 합쳐서 최대 50점.

3단계 — 통합 점수 (0~500점)

모든 영역의 점수 + 흐름 점수 + SLA + Sorter + MAXCAPA → 통합 점수.
이 점수로 위험도 등급 결정.

5R-A 룰 — 반송시간 (얼마나 느려졌나?)

한 문장 요약

"FOUP 한 개 옮기는 데 걸리는 시간이 평소보다 길어졌나?" 를 검사하는 룰.

평소 집에서 회사까지 30분 걸리는데, 오늘 1시간 걸렸다. 길이 막혔다는 신호.

5-1. 측정하는 컬럼 (영역마다 다름)

영역마다 측정 방식이 다르고 임계값도 다릅니다:

영역 측정 컬럼 임계 (분) 의미
M16HUB M16HUB.QUE.TIME.AVGTOTALTIME1MIN 9.0 HUB 의 1분 평균 반송시간
M14 M14.QUE.LOAD.AVGLOADTIME1MIN 3.3 M14 의 1분 평균 적재시간
M14B M14B.QUE.TIME.AVGTOTALTIME1MIN 5.0 M14B 의 1분 평균 반송시간
M16A M16A.QUE.LOAD.AVGLOADTIME1MIN 3.2 M16A 의 1분 평균 적재시간
M16B M16B.QUE.LOAD.AVGLOADTIME1MIN 3.5 M16B 의 1분 평균 적재시간

영역마다 임계가 다른 이유: M16HUB 는 본래 처리량이 많아서 평소 반송시간이 길고, M14 는 짧습니다. 같은 9분이라도 영역에 따라 의미가 다릅니다.

5-2. R-A 는 두 가지로 나뉩니다

① RA (절대) — +10점

조건: 지난 10분 동안 한 번이라도 임계값을 넘었나?

시각: 분1 분2 분3 분4 분5 분6 분7 분8 분9 분10 M16HUB: 5.0 6.2 5.8 9.5 4.0 3.8 5.0 6.1 4.9 5.2 ↑ 여기서 9.5 ≥ 9.0 한 번 → RA 발동! → +10점

② RA_sus (지속) — +5점

조건: 지난 5분 동안 3번 이상 (TH_RA × 0.67) 을 넘었나?

M16HUB 의 경우: 9.0 × 0.67 = 6.03 분 이 sustained 임계.

시각: 분1 분2 분3 분4 분5 M16HUB: 6.5 5.8 6.1 6.2 4.5 ↑ ↑ ↑ (3번이 ≥ 6.03 → RA_sus 발동! → +5점)
왜 두 가지로 나누나?
• RA (절대) = 갑작스런 spike. "9분 한 번"은 일시적 사고일 수 있음.
• RA_sus (지속) = 만성 정체. "6분이 3번"이 더 무서움 — 계속 막혀있다는 뜻.

5-3. 두 조건 모두 발동하면?

RA + RA_sus 둘 다 ON → +10 + +5 = +15점 가산.

6R-B 룰 — 큐 누적 (얼마나 쌓이나?)

한 문장 요약

"처리되지 않은 FOUP 이 시간이 지나면서 쌓이고 있나?" 를 검사하는 룰.

마트 계산대 줄 길이가 30분 전엔 5명이었는데 지금 50명이라면 → 점원이 처리를 못 따라가는 상황. 30분 동안 +45명 쌓임.

6-1. 측정하는 컬럼 (영역마다 다름)

영역 측정 컬럼 (큐 카운트) 30분 임계 10분 임계
M16HUB M16HUB.QUE.M14TOM16.MESCURRENTQCNT +100 +30
M14 M14.QUE.ALL.3F_TO_HUB_JOB +80 +24
M14B M14B.QUE.ALL.7F_TO_HUB_JOB +150 +45
M16A M16A.QUE.ALL.6F_TO_HUB_JOB +80 +24
M16B M16B.QUE.ALL.10F_TO_HUB_JOB +30 +10
M16 M16.QUE.SFAB.SENDQUEUETOTAL +20 +10

10분 임계 = max(10, 30분 임계 × 0.3) 으로 자동 계산.

6-2. R-B 도 두 가지로 나뉩니다

① RB (30분 누적) — +10점

조건: 현재 큐 카운트 - 30분 전 큐 카운트 ≥ 임계.

[M16HUB 예시] 30분 전 시점: M14TOM16 큐 = 50 개 ↓ 30분이 지나서 ↓ 현재 시점: M14TOM16 큐 = 155 개 변화: 155 - 50 = +105 개 ≥ 100 (임계) → RB 발동! M16HUB 영역에 +10점
사용자 예시 M14TOM16.MESCURRENTQCNT 의 30분 변화량이 100보다 크면 → R-B 의 30분 누적 룰 입니다. (R-A 가 아님 주의)

② RB_fast (10분 폭주) — +5점

조건: 현재 큐 - 10분 전 큐 ≥ (30분 임계 × 0.3).

[M16HUB 예시] 10분 전: M14TOM16 큐 = 120 개 현재: M14TOM16 큐 = 155 개 변화: 155 - 120 = +35 개 ≥ 30 (임계) → RB_fast 발동! +5점
왜 30분과 10분 따로?
• RB (30분) = 점진적 데드락. 천천히 쌓이는 만성 정체.
• RB_fast (10분) = 폭주. Bridge OHT Error 같은 급변 사고.
같은 큐 누적이지만 원인이 다르니 따로 잡습니다.

7R-C 룰 — 반전 흐름 (모양이 깨졌나?)

한 문장 요약

"전체 큐 합계는 그대로인데, 특정 리프터에 몰리거나 한쪽 CNV 가 막혔나?" 를 검사하는 룰.

도시 전체 차량 수는 똑같지만, 한 도로에만 차가 몰리고 다른 도로는 비어있다면 → 어딘가 사고가 있다는 신호.

7-1. M16HUB 리프터 10대 반전 — +8점

M16HUB 에는 리프터가 10대 있습니다:

6ABL6031, 6032M16A 3F 대응
6ABL0111, 0112, 0121, 0122M16EUV 대응

발동 조건 (두 가지 모두 만족):

  1. 20분 전 대비 10대 큐 합계가 감소 (rc_trend < 0)
  2. 개별 큐가 증가한 리프터가 2대 이상 (rev_count ≥ 2)

의미: 전체적으로 큐는 줄었는데, 특정 리프터들에 큐가 쏠리고 있음. 흐름이 한 방향으로 편향된 상태.

왜? 한쪽 CNV 가 막혔거나, 한쪽으로만 작업이 몰리고 있다는 신호.

8R-D 룰 — Fab 적체 (공간이 부족한가?)

한 문장 요약

"저장 공간이 거의 다 차서 새 FOUP 을 못 받는 상태인가?" 또는 "처리 장비가 풀가동되어 여유가 없나?" 를 검사하는 룰.

주차장이 99% 차면 새 차가 들어올 수 없습니다. 마찬가지로 FAB 저장 공간이 차면 새 FOUP 을 받을 수 없습니다.

8-1. M16HUB Fab 적체 — +7점

발동 조건 (둘 중 하나라도):

  1. M16HUB.STRATE.ALL.FABSTORAGERATIO ≥ 25.0%
    → M16HUB 의 fab 저장 사용률이 평소 분포 (p95) 의 25% 넘음
  2. M16HUB.STRATE.STB.3F_STORAGE_UTIL ≥ 99.0%
    → STB (임시 저장소) 가 거의 꽉 참
[정상 상황] FABSTORAGERATIO = 15% STB_UTIL = 80% → R-D 비발동, 0점 [적체 상황] FABSTORAGERATIO = 28% ≥ 25.0 또는 STB_UTIL = 99.5% ≥ 99.0 → R-D 발동! M16HUB 영역에 +7점

8-2. 다른 영역 OHT 가동률 — +7점

대상: M14, M14B, M16A, M16B

발동 조건: *.QUE.OHT.OHTUTIL ≥ 95.0%

[정상 상황] M14.OHT.OHTUTIL = 75% → R-D 비발동 [과부하 상황] M14.OHT.OHTUTIL = 97.3% ≥ 95.0 → R-D 발동! M14 영역에 +7점
왜 임계 25 / 99 / 95?
• 25% — M16HUB 평소 분포의 상위 5% (p95). 평소 흔히 보는 값이 아님.
• 99% — STB 가 사실상 꽉 찬 시점 (마지막 1대 공간만 남음).
• 95% — OHT 가 신규 작업 받을 여유가 거의 없는 시점.

한 영역 (예: M16HUB) 에서 발동된 모든 룰의 점수를 합산. 최대 50점에서 클리핑.

RA_susR-A 지속 임계 발동+5
RBR-B 30분 누적 발동+10
RB_fastR-B 10분 폭주 발동+5
RCR-C 반전 발동+8
RDR-D 적체 발동+7

SORT소터 대기 큐 임계 위반+3

area_score = min(50, RA + RA_sus + RB + RB_fast + RC + RD + SLA + SORT + MAXCAPA)
[M16HUB 예시] RA = +10 (발동) RA_sus = +5 (발동) RB = +10 (발동) RB_fast = +5 (발동) RC = +8 (발동) RD = +7 (발동) SLA = +5 (발동) SORT = 0 (미발동) MAXCAPA = 0 (미발동) ──────────────────── 합계 = 50 min(50, 50) = 50점 ← M16HUB area_score
영역 점수의 최대값은 50 으로 고정. 어떤 영역도 50을 넘지 않습니다.

10통합 점수 만들기 (0~500점)

모든 영역의 점수 + 흐름 점수 + SLA + Sorter + MAXCAPA → 전체 시스템 점수.

통합 점수 공식 (5 항 합산) 
unified_risk_score = min(500,
    layer1_total     # 8 영역 점수 합산
  + flow_score        # 10 흐름 노드 점수
  + sla_score         # SLA 발동 영역 수 × 5
  + sorter_score      # 소터 발동 영역 수 × 3
  + mc_score          # MAXCAPA 변경 수 × 10
)

10-1. layer1_total — 영역 점수 합산

8 영역 (M16HUB, M14, M14B, M16A, M16B, M16, M16_PKT, M16_WT) 의 area_score 합산.

최대 = 50 × 8 = 400점

10-2. flow_score — 흐름 노드 점수

영역 간 흐름을 보는 10 개 노드. 각 노드의 현재 값 / 평소 값 (ratio) 으로 판정:

ratio등급점수

12Stage 판정 — 진짜 사건인가?

점수 (degree) 와 별개로 Stage 가 있습니다. Stage 는 "이게 진짜 사건인가?" 라는 yes/no 판정입니다.

Stage 0 — 이벤트없음

모든 룰 정상.

Stage 1 — 조기경보

R-A 만 발동. 시간 지연 신호 (한 가지) 만.

조건: 어느 영역이든 RA OR RA_sus

Stage 2 — 주의보

R-B 만 발동. 큐 누적 신호 (한 가지) 만.

조건: 어느 영역이든 RB OR RB_fast

Stage 3 — ⭐확정 사건 ★

3 가지 종류 신호가 동시에 발동.

조건 (모두 만족):

  1. any_ra = 어느 영역이든 R-A 발동 (시간)
  2. any_rd_or_sla 또는 any_rc = R-D 또는 SLA 또는 R-C 발동 (공간/패턴)
  3. any_rb 또는 any_flow_severe = R-B 또는 흐름 위험/심각 (누적)
Stage 3 공식 
S3 = any_ra
   AND (any_rd_or_sla OR any_rc)
   AND (any_rb OR any_flow_severe)
왜 S3 가 3중 조건?
단일 룰 1개만 발동했다고 사건이라고 판정하면 위양성 (false alarm) 이 너무 많아집니다.
시간 + 공간/패턴 + 누적 — 3 가지 시각이 모두 일치할 때만 "확정 사건" 으로 판정합니다.

14자주 묻는 질문

Q1. 점수 75 가 등급 "경계" 인데 Stage 3 도 확정이면 모순 아닌가?
아닙니다. 점수 (degree, 정도) 와 Stage (사건 발생 여부) 는 서로 다른 차원입니다.
• 점수 = 정량적 위험도 (얼마나 위험한가)
• Stage = 정성적 사건 판정 (3가지 시각이 동시 일치하는가)
Stage 3 이지만 점수가 낮을 수 있고, 점수가 높지만 Stage 1 일 수도 있습니다. 둘 다 봐야 합니다.
Q2. 임계값들 (9.0, 100, 0.67 등) 은 어떻게 정해졌나?
2026년 3월 24일 ~ 4월 30일 사이의 실제 운영 데이터 분포를 분석:
• TH_RA = p95 (95 백분위) 또는 운영자 경험적 임계
• TH_RB = 사건 발생 직전 30분 평균 증가량
• TH_RC_REVERSE = 2 (단일 1대로는 노이즈, 3대 이상이면 너무 보수적)
• TH_RD = 운영 매뉴얼 기준치
Q3. 영역마다 임계값이 다른 이유?
각 영역의 평소 분포 (baseline) 가 다릅니다. M16HUB 는 본래 처리량이 많아 평소 반송시간 3~5분이 정상이지만, M14 는 1~2분이 정상. 같은 9분이라도 영역에 따라 의미가 완전 다르므로 영역별 임계를 따로 두었습니다.
Q4. R-A 만 발동하면 점수가 얼마인가?
M16HUB 만 R-A 발동 가정:
• RA +10 → area_score 10
• layer1_total = 10
• flow_score = 0 (가정)
• 다른 점수 = 0
• unified_risk_score = 10 → 등급 "정상" (< 30)
• Stage 1 (조기경보)
Q5. 같은 룰이 여러 영역에 동시 발동하면?
영역마다 따로 점수 매김. 예: R-A 가 M16HUB + M14 에 동시 발동 → M16HUB area_score +10, M14 area_score +10. 두 영역 점수 모두 layer1_total 에 합산됨.
Q6. unified_risk_score 가 500 을 넘으면?
넘지 않습니다. min(500, ...) 으로 클리핑. 이론상 최대 500.
Q7. Stage 3 인데 등급이 "정상" 일 수 있나?
이론적으로 가능하지만 실제로는 극히 드뭅니다. Stage 3 조건이 만족되면 보통 점수가 50+ 이상으로 올라가 "관심" 이상 등급이 됩니다. 만약 Stage 3 + 점수 낮으면 → 룰 임계가 너무 낮게 설정되었다는 신호 (재조정 필요).
Q8. 매분 자동으로 모든 점수가 계산되나?
예. run_ml.py 가 매분 5초에 hubroom_predictor 를 실행하여 발동이벤트.csv 에 한 행 기록. 동시에 Logpresso test_table3 에 적재. 운영자는 CSV 또는 Logpresso 대시보드에서 실시간 확인.
