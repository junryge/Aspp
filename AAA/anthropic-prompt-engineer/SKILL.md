# Anthropic 프롬프트 엔지니어링 스킬

사용자의 요구사항을 분석하여 Anthropic Claude에 최적화된 고품질 프롬프트를 설계하고 생성하는 전문 스킬이다.

## 공통 규칙

- 반드시 **한국어**로 설명 및 안내
- 생성되는 프롬프트 내용은 사용자 요청에 따라 한국어 또는 영어
- 마크다운 형식 사용 (제목, 리스트, 코드 블록)
- 프롬프트 코드는 ```markdown 또는 ```text 블록으로 감싸기
- 각 프롬프트에 **사용 목적**과 **기대 효과**를 명시

---

## Anthropic 핵심 원칙 (모든 모드에서 반드시 적용)

### 1. 명확한 작업 정의
- 모호한 표현 금지. "잘 해줘" 대신 구체적 행동을 지시
- 입력과 출력을 명시적으로 정의
- 성공 기준을 측정 가능하게 작성

### 2. XML 태그 활용
- 구조화된 입력/출력에 XML 태그를 적극 사용
- 태그명은 의미를 담아 작성: `<instructions>`, `<context>`, `<examples>`, `<output_format>`
- 중첩 태그로 계층 구조 표현

### 3. 역할(페르소나) 부여
- 시스템 프롬프트에서 전문가 역할을 명확히 지정
- 역할에 맞는 어조, 전문성 수준, 관점을 구체화
- "당신은 ~입니다" 대신 "당신은 ~전문가다. ~을 수행하라" 형식 사용

### 4. 예시(Few-shot) 제공
- 최소 2-3개의 입출력 예시를 포함
- 예시는 다양한 케이스를 커버 (정상, 엣지, 예외)
- `<example>` 태그로 감싸서 구조화

### 5. 사고 과정(Chain of Thought)
- 복잡한 추론이 필요한 작업에 단계별 사고 유도
- `<thinking>` 태그 안에서 추론 과정을 작성하도록 지시
- 최종 답변은 `<answer>` 태그로 분리

### 6. 출력 형식 명세
- JSON, 테이블, 마크다운 등 원하는 출력 형식을 명시
- 필드명, 타입, 필수 여부를 구체적으로 정의
- 출력 예시를 함께 제공

### 7. 제약 조건과 엣지 케이스
- "~하지 마라" 형태로 금지 사항을 명시
- 입력이 부족하거나 잘못된 경우의 처리 방법을 지정
- 확신할 수 없는 경우 "모른다"고 답하도록 지시

## 생성 모드

### [SYSTEM] 시스템 프롬프트 생성

사용자가 "시스템 프롬프트", "system prompt" 작성을 요청하면:

1. **목적 파악**: 어떤 AI 어시스턴트를 만들려는지 확인
2. **역할 정의**: 전문가 페르소나를 구체적으로 설정
3. **행동 규칙**: 반드시 지켜야 할 규칙을 나열
4. **금지 사항**: 하지 말아야 할 행동을 명시
5. **출력 형식**: 응답 포맷을 정의
6. **톤/매너**: 어조와 커뮤니케이션 스타일 지정

출력 구조:
```text
[역할 정의]
당신은 {역할}이다. {전문 분야}에 대한 깊은 지식을 보유하고 있다.

[핵심 임무]
- {임무 1}
- {임무 2}

[행동 규칙]
1. {규칙 1}
2. {규칙 2}

[금지 사항]
- {금지 1}
- {금지 2}

[출력 형식]
{형식 설명}

[톤/매너]
{어조 설명}
```

### [TASK] 작업 프롬프트 생성

사용자가 "프롬프트 만들어줘", "작업 프롬프트" 등을 요청하면:

1. **작업 분석**: 사용자가 원하는 결과물을 정확히 파악
2. **컨텍스트 설정**: 배경 정보와 제약 조건 정리
3. **단계 분해**: 복잡한 작업을 단계별로 분해
4. **입출력 정의**: 입력 데이터와 기대 출력을 명시
5. **예시 포함**: 구체적인 입출력 예시 2-3개 작성
6. **엣지 케이스**: 예외 상황 처리 방법 포함

출력 구조:
```xml
<task_description>
{작업 목적과 배경을 구체적으로 기술}
</task_description>

<instructions>
1. {단계 1}
2. {단계 2}
3. {단계 3}
</instructions>

<context>
{배경 정보, 도메인 지식, 제약 조건}
</context>

<examples>
<example>
<input>{입력 예시 1}</input>
<output>{출력 예시 1}</output>
</example>
<example>
<input>{입력 예시 2}</input>
<output>{출력 예시 2}</output>
</example>
</examples>

<output_format>
{출력 형식 상세 명세}
</output_format>

<constraints>
- {제약 1}
- {제약 2}
</constraints>
```

### [SKILLMD] SKILL.md 형식 프롬프트 생성

사용자가 "스킬 프롬프트", "SKILL.md 형식으로", "우리 MD처럼" 등을 요청하면:

1. **스킬 목적 파악**: 어떤 기능의 스킬을 만들려는지 확인
2. **트리거 키워드 도출**: 스킬이 활성화될 키워드/문맥 정의
3. **YAML 프론트매터 작성**: name, description 필수 포함
4. **공통 규칙 정의**: 언어, 형식, 참조 규칙 등
5. **모드 설계**: 기능별 모드를 `[TAG]` 형태로 분리
6. **각 모드별 단계**: 번호 매긴 구체적 실행 단계 작성
7. **응답 형식**: 출력 포맷 정의
8. **연동 스킬**: 관련 스킬 목록 추가

출력 구조:
```markdown
---
name: {스킬명-소문자-하이픈}
description: >
  {한국어 설명. 스킬이 하는 일 + 활성화 트리거 문맥을 구체적으로.}
metadata:
  author: {작성자}
  version: "1.0.0"
  tags:
    - {태그1}
    - {태그2}
---

# {스킬 한국어 제목}

{스킬 소개 한 문단}

## 공통 규칙
- 반드시 **한국어**로 {작업 내용}
- {형식 규칙}
- {참조 규칙}

## 분석/실행 모드

### [{TAG1}] {모드1 제목}
사용자가 "{트리거 표현}"을 요청하면:
1. **{단계1 제목}**: {구체적 행동}
2. **{단계2 제목}**: {구체적 행동}
3. **{단계3 제목}**: {구체적 행동}

### [{TAG2}] {모드2 제목}
사용자가 "{트리거 표현}"을 요청하면:
1. **{단계1}**: {행동}
2. **{단계2}**: {행동}

## 응답 형식
{출력 포맷 정의}

## 연동 가능 스킬
- `{스킬id}`: {설명}
```

**SKILL.md 작성 시 핵심 규칙:**
- name은 소문자, 숫자, 하이픈만 사용 (regex: `^[a-z0-9-]+$`)
- description은 "푸시형"으로 — 트리거 문맥을 적극적으로 나열
- 전체 500줄 이내
- 지시문은 명령형으로 작성 ("~하라", "~마라")
- 모드 태그는 대괄호 `[TAG]` 형식

### [OPTIMIZE] 프롬프트 최적화

사용자가 "프롬프트 개선", "프롬프트 최적화", "이 프롬프트 고쳐줘" 등을 요청하면:

1. **현재 프롬프트 분석**:
   - 명확성: 지시가 모호한 부분 식별
   - 구조: XML 태그, 섹션 분리 여부 점검
   - 예시: few-shot 예시 유무 및 품질 평가
   - 제약: 금지 사항, 엣지 케이스 처리 여부
   - 출력 형식: 명세 구체성 평가

2. **개선 항목 도출**: 각 항목에 **우선순위** 표시
   - 🔴 높음: 결과 품질에 직접 영향
   - 🟡 중간: 일관성/안정성에 영향
   - 🟢 낮음: 가독성/유지보수에 영향

3. **개선된 프롬프트 제공**:
   - 개선 전/후 비교 (diff 형식)
   - 각 변경의 이유 설명

4. **테스트 케이스 제안**: 개선된 프롬프트를 검증할 입력 3-5개

### [CHAIN] 체인 프롬프트 설계

사용자가 "프롬프트 체인", "다단계 프롬프트", "파이프라인 프롬프트" 등을 요청하면:

1. **전체 작업 분석**: 최종 목표와 중간 산출물 정의
2. **단계 분리**: 각 단계를 독립적인 프롬프트로 분리
3. **입출력 연결**: 이전 단계 출력이 다음 단계 입력이 되도록 설계
4. **각 단계 프롬프트 작성**: 위 [TASK] 모드 형식으로 각 단계 작성
5. **흐름도 제공**: 전체 체인을 시각적으로 표현

출력 구조:
```text
[체인 흐름도]
Step 1: {작업명} → 출력: {산출물}
  ↓
Step 2: {작업명} → 입력: {Step1 산출물} → 출력: {산출물}
  ↓
Step 3: {작업명} → 입력: {Step2 산출물} → 출력: {최종 결과}

[Step 1 프롬프트]
...

[Step 2 프롬프트]
...

[Step 3 프롬프트]
...
```

## 프롬프트 품질 체크리스트

모든 모드에서 생성된 프롬프트는 아래 항목을 충족해야 한다:

| 항목 | 기준 |
|------|------|
| 명확성 | 모호한 표현 없이 구체적 행동을 지시하는가? |
| 구조화 | XML 태그 또는 마크다운으로 구조가 분리되어 있는가? |
| 역할 | 적절한 페르소나/전문성이 부여되었는가? |
| 예시 | 2개 이상의 입출력 예시가 포함되어 있는가? |
| 출력 형식 | 원하는 출력 포맷이 명세되어 있는가? |
| 제약 조건 | 금지 사항과 엣지 케이스 처리가 있는가? |
| 길이 | 불필요한 반복 없이 간결한가? |
| 테스트 | 다양한 입력에 대해 일관된 결과를 내는가? |

## Anthropic 고급 기법 참조

### 프리필(Prefill) 기법
- Assistant 턴의 첫 부분을 미리 채워서 출력 형식을 강제
- 예: `{"result":` 로 시작하게 하여 JSON 출력 유도

### 롱 컨텍스트 전략
- 중요한 지시를 프롬프트의 **처음과 끝**에 배치
- 대량 문서 참조 시 `<document>` 태그로 감싸기
- 핵심 지시는 문서 뒤에 반복 강조

### 할루시네이션 방지
- "확실하지 않으면 '잘 모르겠습니다'라고 답하라"
- "제공된 문서에 없는 내용은 답하지 마라"
- 인용/출처 명시를 필수로 지정

### 일관성 확보
- 온도(temperature) 0에 가깝게 설정 권장
- 출력 스키마를 엄격히 정의
- 부정형("~하지 마라")보다 긍정형("~하라")을 우선 사용
