AMHS · MCS 연동 설계

에이전트 ↔ MCS
MCP 인터페이스 설계

AI 에이전트가 물리 반송장비(OHT/AGV)를 다루는 MCS에 안전하게 붙기 위한 인터페이스 규격. 읽기와 제어를 완전히 분리하고, 제어는 MCS 정식 채널만 경유한다.

Read plane — Oracle SELECT Control plane — 커맨드 스테이징 이중 인터락 streamable-http : 8010
단 하나의 원칙

MCP는 MCS 상태를 직접 바꾸지 않는다. "요청"만 넣고, 판단·검증·실행은 전부 MCS가 한다.

MCP가 OHT 커맨드 테이블에 직접 INSERT/UPDATE 하면 MCS 상태머신과 인터락이 무시된다 → carrier 유실, deadlock 발생. 절대 금지. MCP는 접수창구, MCS는 최종 판단자다.
00

전체 레이어 구조읽기 경로와 제어 경로가 서버 안에서부터 갈라진다

에이전트는 MCP 서버 하나에만 붙는다. 서버 내부에서 Read 어댑터와 Control 어댑터가 나뉘고, 각각 목적지가 다르다. Read는 read-only Oracle 계정으로 직행, Control은 MCS 커맨드 스테이징 테이블로만 간다. MCS는 그 테이블을 폴링해서 자기 인터락을 태워 실행한다.

Read plane · 안전 Control plane · 주의 인터락 — 색이 곧 의미다. 차가운색=읽기, 따뜻한색=제어.
01

Read plane (Phase 1)read-only 계정으로 SELECT만 — 지금 하던 방식 그대로

부작용이 전혀 없는 조회 계층. Oracle read-only 계정으로 SELECT만 래핑한다. 결과는 structuredContent로 반환하되 camelCase를 지킨다(스모크 테스트에서 물렸던 그 부분). 툴은 목적별로 잘게 쪼갠다.

get_carrier_location(carrier_id)
carrier 현재 위치 / 상태
get_oht_status(vehicle_id | zone)
차량 상태 / 가동률
get_queue_depth(idc)
IDC 큐 적재량 (M16HUB.QUE 등 ~100 IDC)
get_route(from, to)
유효 경로 존재 여부
get_command_history(carrier_id, n)
최근 명령 이력 n건

여긴 쿼리 래핑이라 리스크가 없다. Read-only만 쓸 거면 배치는 RHEL 8.10으로 충분하다.

02

Control plane (Phase 2)여기가 진짜 승부처 — 한 방에 실행 안 시킨다

모든 제어 툴은 아래 6단계 파이프라인을 강제로 태운다. dry_run이 기본값이고, 실제 실행은 confirm=True일 때만 일어난다. 실행은 스테이징 테이블 INSERT로 끝나고 절대 block하지 않는다 — carrier 이동은 초 단위라 비동기다.

# 제어 툴 시그니처 — dry_run 기본 / idempotent / 추적 가능
def transfer_carrier(
    carrier_id: str,
    to_dest:   str,
    priority:  int = 5,
    confirm:   bool = False,   # 기본 dry_run — 실행 안 함
    idem_key:  str  = None,    # 재시도 중복 차단 키
    agent_id:  str  = None     # 누가 쐈나 (추적)
) -> dict:
1

validate 순수 검증

carrier 실존? from에 진짜 있나? to 수용 가능? 유효 route 있나? 하나라도 깨지면 isError=true로 tool-level 에러 반환(스모크 때 잡은 그 플래그). 여기서 걸러야 MCS까지 안 간다.

2

gate 1차 인터락

rule-based deadlock 로직(R-A' / R-B / R-C' / R-D)과 queue 임계를 그대로 재활용. deadlock·surge 감지되면 제어명령 자체를 막는다. 이게 MCP 방어선이다.

3

confirm 분기 게이트

False → 실행 안 하고 dry_run 결과만: 예상 route, 예상 소요, 리스크 스코어(hub_high×3 + storage_util_critical×2 …). 에이전트가 이걸 보고 판단.
True → 실제 submit으로 진행.

4

submit 스테이징 INSERT

MCS 커맨드 스테이징 테이블에 INSERT. DB 직접쓰기와 다른 이유 — 이 테이블은 MCS가 폴링하는 정식 입력구고, MCS가 픽업해서 자기 인터락(2차)을 태워 실행한다. MCP는 명령서를 접수창구에 넣을 뿐 직접 실행하지 않는다. 반환은 command_id 하나. block 없음.

5

verify 별도 폴링 툴

get_command_status(command_id) 로 상태 폴링. 이동이 끝날 때까지 툴이 기다리지 않는다.

# submit — idem_key UNIQUE 제약으로 중복 명령 원천 차단
INSERT INTO mcs_cmd_staging
  (idem_key, cmd_type, carrier_id, dest, priority, agent_id, submit_ts, status)
VALUES (:idem_key, 'TRANSFER', ..., 'SUBMITTED')
-- 같은 idem_key 재요청 → 기존 command_id 반환, 재실행 안 됨

명령 상태머신 — MCP 게이트를 통과했는데 MCS가 FAILED를 뱉으면 그게 정상이다. MCS가 최종 판단자니까. MCP는 요청, MCS는 결정.

FAILED는 MCS 2차 인터락이 막은 정상 결과. MCP gate(1차)를 통과해도 MCS(2차)가 최종 거부권을 갖는다.
03

권한 · HITL · 감사누가 무슨 명령을 언제 쐈는지 전부 추적된다

툴마다 권한 레벨 태그를 박는다. critical 등급은 skill factory의 HITL(human-in-the-loop) 게이트를 그대로 붙여서, 사람 승인이 없으면 SUBMITTED에서 멈춰 있게 한다.

레벨범위대상 예시
read전 에이전트 허용위치·큐·상태 조회 전부
control화이트리스트 에이전트만 (agent_id 토큰 검증)단건 transfer, priority 변경
criticalHITL 강제 — 사람 승인 필수batch move, abort_all, zone 통제

감사 — 모든 제어 요청은 이중 기록한다. ① JSONL(Hermes 패턴): who / what(tool+args) / when / result. ② DB audit 테이블: 나중에 역추적용. idem_key까지 같이 남겨서 재시도 흐름도 추적 가능하게.

04

분기 : MCS가 SECS/GEM만 받을 때스테이징 테이블을 안 열어주는 경우

Alternative path

파이프라인은 똑같다. 통로만 바뀐다.

MCS가 커맨드 스테이징 테이블을 안 열어주고 SECS/GEM으로만 제어를 받으면, MCP 서버가 secsgem을 물고 HSMS 세션을 유지하며 S2Fx 커맨드를 송신하는 구조로 간다. 단 —

  • 6단계 파이프라인(validate→gate→confirm→submit→verify)은 그대로 유지된다.
  • 차이는 ④submit이 INSERT 대신 SECS 메시지 송신, ⑤verify가 S6Fx 이벤트/응답 수신으로 바뀌는 것뿐.
  • 대신 HSMS 세션 관리(재접속·timeout)를 MCP 서버가 떠안아야 해서 스테이징 방식보다 무겁다.

결론 — 가능하면 스테이징 테이블 방식이 낫다. 세션 상태를 안 물고, MCS 인터락을 자연스럽게 타니까.
