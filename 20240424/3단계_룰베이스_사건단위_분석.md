# 3단계 데드락 룰베이스 사건 단위 추출 스크립트 분석

> **파일**: `3단계_룰베이스_사건단위.py`
> **목적**: 운영 환경 시뮬레이션 — STAR CSV 한 줄씩 처리하면서 과거 90분 데이터만 보고 3단계 데드락 위험 사건을 사건 단위로 추출

---

## 1. 전체 구조 한눈에 보기

```
STAR CSV (1분 단위)
   ↓
[iter_star_rows]  한 줄씩 읽어서 (시각, 메트릭) 만들어 yield
   ↓
[90분 슬라이딩 윈도우]  t1_window / m14_window / lft_window  (deque maxlen=90)
   ↓
[evaluate_rules]  R-A' / R-B / R-C' 평가 → S1/S2/S3 판정
   ↓
[IncidentTracker FSM]  IDLE ↔ IN_INCIDENT 상태머신
   ↓
[CSV 출력]  날짜별 발동이벤트.csv (매 분 1행) + 사건단위.csv (S3만)
```

핵심 설계 의도는 **"운영처럼 동작하는지 검증"** — 일괄 분석이 아니라 매 분 새 데이터가 들어오는 순간 과거 90분만 보고 판정하는 시뮬레이션. 그래서 실시간 감시 모드(`--watch`)도 같은 코드 경로를 쓴다.

## 2. 룰 정의 (3차 검증 완료 — 변경 금지)

### 2.1 R-A' : 반송시간 절대 임계

```python
TH_RA_VALUE = 9.0  # 분
recent_t1 = t1_list[-10:]            # 최근 10분
ra_count = sum(1 for v in recent_t1 if v >= 9.0)
ra_trig = ra_count >= 1
```

**의미**: 최근 10분 안에 `AVGTOTALTIME1MIN`이 9분 이상으로 튄 적이 1회라도 있으면 R-A' 발동.

**왜 9분인가**: 정상 반송시간 대비 매우 길다는 신호 — OHT가 정체됐거나 우회 라우팅 중이라는 뜻.

### 2.2 ra_sustained : 반송시간 지속 임계

```python
TH_RA_SUSTAINED_VALUE = 6.0
TH_RA_SUSTAINED_COUNT = 3
last5 = recent_t1[-5:]
ra_sustained = sum(1 for v in last5 if v >= 6.0) >= 3
```

**의미**: 최근 5분 중 3분 이상이 6분 임계를 넘으면 발동.

**왜 따로 두나**: 9분처럼 강하진 않지만 6분대가 지속되면 그것도 위험. 단발성 스파이크와 만성 정체를 구분하려는 장치.

### 2.3 R-B : FAB간 큐 폭증 (30분)

```python
TH_RB_DIFF_30 = 100
rb_diff = m14_list[-1] - m14_list[-31]   # 30분 전 대비
rb_trig = rb_diff >= 100
```

**의미**: M14→M16 큐가 30분 전보다 100개 이상 늘었으면 발동.

**왜 M14→M16만 보나**: 메모리에 있는 패턴 그대로 — 이게 가장 데드락에 잘 드러나는 트랜스포트 흐름이라서.

### 2.4 rb_fast : FAB간 큐 단기 폭증 (10분)

```python
TH_RB_DIFF_10 = 30
rb_diff_10 = m14_list[-1] - m14_list[-11]
rb_fast = rb_diff_10 >= 30
```

**의미**: 30분짜리 R-B는 늦을 수 있어서 10분짜리 fast 채널 추가.

### 2.5 R-C' : 리프터 정체 + 역증가

```python
TH_RC_REVERSE = 2
rc_trend = sum(now_l.values()) - sum(prev_l.values())   # 20분 전 대비 합 변화
rev_lids = [lid for lid in now_l if now_l[lid] > prev_l.get(lid, 0)]
rev_count = len(rev_lids)
rc_trig = rc_trend < 0 and rev_count >= 2
```

**의미**: 전체 리프터 합은 줄었는데(작업이 빠져나간 건데) 동시에 개별로 늘어난 리프터가 2개 이상 있으면 발동.

**왜 이게 데드락 신호인가**: 합은 줄어드는데 일부 리프터만 적체된다는 건 **로컬 정체가 풀리지 않고 일부 노드에 쌓이고 있다**는 뜻. 이게 R-C'의 핵심 통찰.

대상 리프터 (10개):
```
6ABL6011, 6ABL6012, 6ABL6021, 6ABL6022, 6ABL6031, 6ABL6032,
6ABL0111, 6ABL0112, 6ABL0121, 6ABL0122
```

### 2.6 단계 판정

```python
s1 = (ra_count >= 2) or ra_sustained          # 1단계: 조기경보
s2 = rb_trig or rb_fast                       # 2단계: 주의보
s3 = ra_trig and rb_trig and rc_trig          # 3단계: 확정 (AND!)
```

**핵심**: S3는 **AND** 조건이다. 세 가지 다른 종류의 신호(반송시간 + 큐 폭증 + 리프터 정체)가 동시에 발생할 때만 진짜 데드락. 단일 룰로는 false positive가 너무 많아서 AND 조합으로 정밀도를 끌어올린 것.

> 메모에도 있듯 S3 룰은 **2025-04-21 검증 보호** — 절대 변경 금지.

## 3. 슬라이딩 윈도우 처리 디테일

```python
WINDOW_MIN = 90
t1_window  = deque(maxlen=90)   # AVGTOTALTIME1MIN
m14_window = deque(maxlen=90)   # M14→M16 큐 카운트
lft_window = deque(maxlen=90)   # 리프터별 카운트 dict
```

### 3.1 워밍업 처리

```python
if len(t1_window) < 31:
    continue
```

R-B가 30분 전 데이터를 비교해야 하므로 **31개 이상 채워질 때까지 룰 평가를 보류**한다. CSV 시작 첫 30분은 무조건 통과시킴 — 이게 운영 시뮬레이션에서 중요한 이유는 **재시작 직후 30분간 false positive 없게** 하려는 것.

### 3.2 deque의 의미

`deque(maxlen=90)`이라 91번째 데이터 들어오면 가장 오래된 것 자동 폐기. 메모리 효율적이고, 인덱스 `[-1]`/`[-31]`로 깔끔하게 과거 시점 비교 가능.

## 4. IncidentTracker FSM — 사건 단위 합치기

이게 이 스크립트의 **가장 정교한 부분**.

### 4.1 상태 다이어그램

```
        S3 신규 발동
IDLE ───────────────► IN_INCIDENT
 ▲                         │
 │                         │ S3 재발동 (10분 안)
 │                         │ → 같은 사건, refire++
 │                         │ ─┐
 │                         │  │
 │                         │ ◄┘
 │                         │
 │   10분 무 S3            │
 └─────────────────────────┘
```

### 4.2 왜 사건 단위로 합치나

S3가 1분 단위로 깜빡깜빡 발동하는 경우가 많은데, 운영자 입장에선 **"이 시간대 한 번의 사건"**으로 보고 싶다. 그래서:

- S3가 처음 발동 → 사건 시작
- 그 뒤 10분 안에 또 S3 → 같은 사건의 재발동(refire)
- 10분 동안 S3 없음 → 사건 종료

`INCIDENT_END_GAP_MIN = 10`이 이 합치기의 핵심 파라미터.

### 4.3 predict_time — 조기경보 추적

```python
PREDICT_LOOKBACK_MIN = 60
self.early_signals = deque(maxlen=60)

# S3 시작 시
cutoff = t - timedelta(minutes=60)
early = [x for x in self.early_signals if x >= cutoff]
predict_time = min(early) if early else t
```

S3가 확정된 시점에 **그 직전 60분 안에서 최초 S1/S2 발동 시각**을 찾는다. 이게 `predict_time` — 즉 "이 사건은 이 시각부터 알아챌 수 있었다".

`lead_min = (start_time - predict_time)`은 **얼마나 미리 인지 가능했는가**의 정량 지표. 메모에 있는 "조기 경보" 컨셉의 구현체.

### 4.4 사건 통계 누적

```python
c['max_1min']      = max(c['max_1min'], ctx['ra_value'])
c['max_rb_diff']   = max(c['max_rb_diff'], ctx['rb_diff'])
c['max_rev']       = max(c['max_rev'], ctx['rev_count'])
c['rev_lids_union'].update(ctx.get('rev_lids') or [])
c['refire_count']  += 1  # 10분 갭 넘은 재발동 시
```

사건 진행 중 최대치를 추적 → 사건 종료 시 row 변환 단계에서 심각도/원인 분석에 사용.

## 5. 사건 → CSV 행 변환 (`_build_explanation`)

각 사건의 **주요 원인을 자동 판정**하는 휴리스틱:

```python
ra_score = 100 * (ra_val / 9.0  - 1)  if ra_val >= 9.0  else 0  # 9분 기준 초과 %
rb_score = 100 * (rb_val / 100.0 - 1) if rb_val >= 100  else 0  # 100건 기준 초과 %
rc_score = 100 * (rc_val / 2.0  - 1)  if rc_val >= 2    else 0  # 2개 기준 초과 %
```

세 값 중 **최대 점수의 룰을 primary_cause**로 지정. 임계 대비 얼마나 더 심했는지를 정규화한 비교 가능 점수.

영향도 분류:
- `> 50%` → 매우 강함
- `> 20%` → 강함
- `> 0%`  → 보통
- 그 외   → 약함

심각도(`★/★★/★★★`)는 별도:
```python
if refire_count >= 4 or max_1min >= 20: '★★★'
elif refire_count >= 2 or max_1min >= 15: '★★'
elif refire_count >= 1 or max_1min >= 10: '★'
```

**재발동 횟수**와 **반송시간 최대값** 두 축으로 분류 — 짧고 강한 사건과 길게 끄는 사건 둘 다 잡겠다는 의도.

## 6. 출력 CSV 두 종류

### 6.1 발동이벤트 CSV (`<YYYYMMDD>_발동이벤트.csv`)

**매 분 무조건 1행** 기록. 운영 로그 성격.

| 컬럼 | 내용 |
|---|---|
| `file`, `datetime`, `date`, `time` | 식별 정보 |
| `stage`, `stage_name` | 0~3 단계 |
| `prev_stage`, `transition` | 이전 단계, "0→1" 같은 전이 표시 |
| `reason` | 사람이 읽는 사유 |

**왜 이벤트 없는 분도 기록하나**: 운영자가 로그 보다가 "이 시각엔 왜 누락됐지?" 의심 안 들게. **데이터가 들어왔는데 평온했다**는 사실 자체를 기록.

### 6.2 사건단위 CSV (`<YYYYMMDD>_사건단위.csv`)

**S3 사건만** 기록. 의사결정용.

| 컬럼 | 내용 |
|---|---|
| `predict_time`, `start_time`, `end_time` | 조기경보 / 확정 / 종료 |
| `lead_min` | 얼마나 미리 인지 가능했나 |
| `duration_min` | 사건 지속 시간 |
| `refire_count` | 재발동 횟수 |
| `max_1min`, `max_m14_diff`, `max_reverse_lifters` | 룰별 최대치 |
| `primary_cause`, `contrib_breakdown` | 자동 분석 |
| `anomaly_explanation`, `early_warning` | 사람이 읽는 요약 |

### 6.3 날짜별 분리 + append 정책

```python
def append_rows_csv(path, fields, rows):
    new_file = not os.path.exists(path) or os.path.getsize(path) == 0
    with open(path, 'a', encoding='utf-8-sig', newline='') as f:
        w = csv.writer(f)
        if new_file:
            w.writerow(fields)
        for r in rows:
            w.writerow(r)
```

같은 날짜 파일 있으면 추가, 없으면 헤더 쓰고 새로 만든다. 인코딩은 `utf-8-sig` — Excel 한글 깨짐 방지(이미 쓰던 패턴 그대로).

## 7. 두 가지 실행 모드

### 7.1 일괄 처리 (`process`)

```bash
python3 3단계_룰베이스_사건단위.py STAR.csv -o ./out
```

CSV 전체를 한 번에 흐르게 해서 모든 사건/이벤트를 뽑는다. **백테스팅용**.

### 7.2 실시간 감시 (`watch`)

```bash
python3 3단계_룰베이스_사건단위.py STAR.csv --watch --interval 60 -o ./out
```

`os.path.getsize()`로 파일 크기 변화를 감지하고, 변하면 처음부터 다시 읽되 `last_t` 비교로 **새 행만 처리**.

```python
for t, star, _ in iter_star_rows(input_csv):
    if last_t is not None and t <= last_t:
        continue   # 이미 본 행 스킵
    # ... 윈도우 업데이트 + 룰 평가
```

**1분마다 윈도우가 업데이트되고, 평가가 실행되고, 발동이벤트 1행이 즉시 append**된다. 사건이 종료되면 사건단위 CSV에도 즉시 append. Ctrl+C로 종료할 때 `tracker.finalize()` 호출해서 미해소 사건 강제 마감.

**제약**: 매번 파일 처음부터 다시 스캔하는 단순 방식이라 큰 파일에선 비효율. 진짜 운영에선 file offset 추적이나 tail -F 같은 패턴이 더 적절.

## 8. 잠재 이슈 & 개선 포인트

### 8.1 검증된 부분
- ✅ S3 AND 조건 — 04-21 검증 완료, 변경 금지
- ✅ 90분 윈도우 + 31개 워밍업 — R-B 30분 비교 안전
- ✅ 사건 단위 합치기 (10분 갭) — 1분 깜빡임 노이즈 제거
- ✅ utf-8-sig 인코딩 — Excel 호환

### 8.2 검토 필요한 부분

**(1) `iter_star_rows`의 `last_t <= t` 비교**
- 같은 시각 행이 여러 개면 첫 행만 잡고 나머지 무시
- STAR가 진짜 1초 단위면 1분 다운샘플링 누락 가능

**(2) `predict_time` early_signals 중복**
- `early_signals`는 매분 S1/S2 만족 시 추가 → 같은 사건의 S1이 5분 연속이면 5번 들어감
- `min()` 쓰니까 결과는 맞음, 메모리만 약간 비효율

**(3) watch 모드의 파일 처음부터 스캔**
- 24시간 운영하면 점점 느려짐
- 개선안: `last_processed_offset` 저장 후 `f.seek()`로 그 뒤만 읽기

**(4) `rev_lids_union`만 누적되고 출력 안 됨**
- `incident_to_row`에서 사용 안 함 → 의도가 있으면 row에 추가, 없으면 코드 정리

**(5) R-A' 절대 임계 vs sustained 중복 카운트**
- ra_count >= 2 (R-A') 와 ra_sustained 가 동시에 만족 가능
- S1 판정은 OR라 문제 없지만 reason에서 우선순위가 R-A' → 9분짜리 강한 신호가 있을 때 sustained 정보가 묻힐 수 있음

### 8.3 메모 컨텍스트와의 연결

이전에 작업했던 `hid_deadlock_rulebase.py`와 비교하면:
- HID 룰베이스: 14시간 백테스트, HID 37-41 클러스터가 62% 경고 차지, Rule 2/4 비기능
- 본 스크립트: 그것보다 한 단계 위 — **반송시간/큐/리프터 3축 AND 조건**으로 더 강한 정밀도

본 스크립트가 HID 룰베이스보다 false positive가 적을 거라 예상되지만, 반대로 **detection이 늦어질 수 있음** — AND라서 세 조건 다 모일 때까지 기다려야 한다. 그래서 `predict_time` / `lead_min` 같은 조기경보 컨셉이 들어간 거다. **확정은 늦지만 미리 인지는 가능하다**는 운영 메시지.

## 9. 한 줄 요약

**1분 STAR 데이터를 운영처럼 흘리면서 (반송시간 9분+) AND (M14→M16 +100/30분) AND (리프터 합 감소+역증가 2개+) 세 조건을 동시에 만족하는 시점을 데드락 사건으로 잡고, 1분 깜빡임은 10분 갭으로 합쳐 사건 단위로 묶고, 60분 전까지 거슬러 올라가 조기경보 가능 시점을 함께 기록하는 스크립트.**
