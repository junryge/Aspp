01

mmproj가 뭐고 왜 필요한가

GGUF 생태계에선 Gemma-4-31B-it가 두 파일로 쪼개져 배포된다.

  • 메인 GGUF — 텍스트 타워. 언어 모델 본체.
  • mmproj GGUF — 비전 타워(vision encoder + 프로젝터). 이미지를 토큰으로 바꿔 LLM 임베딩 공간에 밀어넣는 역할.
핵심

텍스트만 쓸 거면 mmproj 없어도 된다. 이미지·PDF 인식을 하려면 mmproj가 필수. 빼먹으면 이미지 입력 자체가 안 붙는다.

02

mmproj 파일 어떤 걸 받나

레포 unsloth/gemma-4-31B-it-GGUF 기준. 세 종류 다 올라와 있다.

파일크기용도판정
mmproj-BF16.gguf 1.2 GB 기본. 품질 손실 사실상 없음 추천
mmproj-F16.gguf 1.2 GB BF16 안 되는 구버전 하드웨어면 이거 대안
mmproj-F32.gguf 2.3 GB 무손실. 근데 오버킬 (BF16 대비 실질 차이 X) 서버용

3090(Ampere)은 BF16 네이티브 지원이라 그냥 BF16 쓰면 된다. F32는 VRAM만 1GB 더 먹고 로딩만 느려짐 — V100 클러스터처럼 메모리 남는 서버에서나 의미 있다.

⚠ 함정 — Q8_0 금지

mmproj를 Q8_0로는 못 뽑는다. 비전 텐서 shape이 Q8_0 column alignment랑 안 맞아서 이 모델 quantizer 전부 해당됨. 양자화하려면 Q6_K가 마지노선이고, 어차피 1.2GB니까 그냥 BF16 쓰는 게 맘 편하다.

03

실행 & VRAM 계산

llama-server 실행
llama-server \
  -m gemma-4-31B-it-UD-Q4_K_XL.gguf \
  --mmproj mmproj-BF16.gguf \
  -c 32768 --host 0.0.0.0 --port 8080
3090 24GB 배분

메인 Q4_K_XL ~20GB + mmproj BF16 1.2GB = ~21.2GB. context 길게 잡으면 빡빡하니, 필요하면 메인을 Q3 계열로 내려서 여유 확보.

※ 폐쇄망: 인터넷 되는 데서 메인 GGUF + mmproj-BF16.gguf 둘 다 받아서 같이 반입.

04

이미지 vs PDF — 뭐가 되고 뭐가 안 되나

이미지 (jpg / png)

mmproj로 바로 인식. OCR, 도표 읽기, 설명 다 됨. 프레임 단위 비디오도 가능.

→ 직접 입력 O

PDF

mtmd는 이미지를 받지 PDF를 파싱하진 않는다. 페이지를 이미지로 렌더링해서 넣어야 함.

→ 변환 1단계 필요

즉 PDF는 "이미지로 바꾸는 한 단계"만 끼우면 된다:

05

PDF → 이미지 렌더링

PyMuPDF(fitz)가 제일 간단. 폐쇄망이면 whl 미리 받아서 반입.

page → PNG
import fitz  # PyMuPDF

doc = fitz.open("spec.pdf")
for i, page in enumerate(doc):
    pix = page.get_pixmap(dpi=200)   # 200dpi = OCR 충분
    pix.save(f"page_{i}.png")

그다음 페이지 PNG를 --image로 넣거나, 서버 API로 base64 태워서 전송:

server API (base64)
curl http://127.0.0.1:8080/v1/chat/completions -d '{
  "messages":[{ "role":"user","content":[
    {"type":"image_url","image_url":{"url":"data:image/png;base64,..."}},
    {"type":"text","text":"이 페이지 내용 정리해줘"}
  ]}]
}'
06

실무 주의 — dpi & 디지털 PDF

dpi 타협점: 150~200

  • 너무 낮으면 — 작은 글자 놓친다. 시방서·도면 설명서처럼 텍스트 많은 PDF는 최소 150~200 필요.
  • 너무 높으면 — 이미지 토큰 수 폭증 → context 터짐.
  • → 200dpi가 무난.
✓ 더 나은 길 — 디지털 PDF

텍스트가 선택 가능한(digital) PDF면 vision 태우지 말고 page.get_text()로 텍스트 바로 뽑는 게 훨씬 정확하고 빠르다.

vision OCR은 스캔본이거나 도표·도면이 섞인 페이지에만 쓰는 게 맞다. 안 가리고 전부 이미지로 태우면 느리고 인식률도 오히려 떨어진다.

TL;DR

한 줄 정리

  • mmproj는 BF16 (3090이면 이게 정답) · Q8_0 금지 · 서버면 F32.
  • 이미지 직접 O / PDF는 이미지 변환 거치면 O.
  • 디지털 PDF는 텍스트 추출(get_text)이 vision OCR보다 낫다. 스캔·도면만 vision.
