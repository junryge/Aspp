## 2. 발동이벤트.csv — 핵심 컬럼 (사람이 읽는 앞부분)
| 컬럼 | 뜻 | 해석 |
|---|---|---|
| `datetime` / `date` / `time` | 시각 | 그 분 |
| `stage` / `stage_name` | 단계 (1 조기경보 / 2 주의보 / 3 ⭐확정) | 3단계 = 여러 룰 동시 충족 |
| `prev_stage` / **`transition`** | 직전 단계 / 전이 (예: `2→3`) | **단계가 올라간 순간 = 상승 시점 포착** |
| `unified_risk_score` | 그 분 종합 점수 (1~100, V2 정규화) | 높을수록 위험 |
| `unified_risk_level` | 그 분 등급 (정상~초위험) | §5 등급표 |
| `hot_area` / **`hot_score`** | 가장 강한 영역 / 그 영역 점수 | 시작 영역 + 얼마나 셌나 |
| `affected_areas` | 영향 영역들 (`;` 구분) | 광역인지(4+영역) |
| **`propagation_chain`** | **전파 경로** (영역→영역 시각순) | 정체가 어디서 시작해 어디로 전이됐나 (★보고서 표엔 넣지 않음 — 해석 근거로만) |
| **`flow_signals`** | 흐름 신호 (노드=배수x(등급)) | 어느 반송 흐름이 평소 대비 몇 배인지 |
| **`maxcapa_signals`** | 운영자 CAPA 변경 흔적 | 사람이 설비 한계를 만진 신호 |
| `M16HUB_score`…`M16_WT_score` | **8영역 각 점수** | 영역별 기여 |
| `{영역}_signals` | 영역별 발동 룰 (예: `RA+RC+RD`) | 어떤 룰이 켜졌나 |
| `incident_state` | IDLE / IN_INCIDENT | 이벤트 진행 중인지 |
| `continuity_min` | 이벤트 시작부터 N분째 | 얼마나 오래 |
| `refire_count` | 재발 횟수(진동) | 껐다 켜졌나 |
| `predicted_fault_type` | 장애유형 라벨 | §6 |
| `layer1_total`·`flow_score`·`sla_score`·`sorter_score`·`mc_score` | 점수 5구성 | **이 5개 합 = raw → V2 정규화(÷2.2) = unified_risk_score** |
| `{영역}_pts_{룰}` | 영역×룰별 기여 점수 (45컬럼) | 어떤 룰이 점수 얼마 보탰나 |
| `reason` | 사람이 읽는 사유 요약 | 한 줄 설명 |

**발동이벤트 활용**: `unified_risk_score` 최댓값 행 → 하루 최고 위험 시각. `transition`에 `2→3` 뜬 분 → 확정 진입 순간.

## 3. 사건단위.csv — 핵심 컬럼 (이벤트당 1행)
| 컬럼 | 뜻 | 해석 |
|---|---|---|
| `predict_time` | 룰이 **이벤트를 처음 예측**한 시각 (조기신호 소급) | 가장 빠른 경고 |
| `start_time` | 점수 54+ **확정** 시각 | 실제 확정 |
| `end_time` | 종료 | |
| **`max_risk_time`** | **최고점(몰림) 시각** | ★사건/시각 = 이 값 기준 (시작 아님) |
| **`lead_min`** | **예측 선행시간** = start − predict (분) | 얼마나 미리 맞췄나 (예: 60분 미리) |
| `duration_min` | 지속 시간 | 이벤트 길이 |
| `refire_count` | 재발생 횟수(진동) | 0이면 단발 |
| `max_risk_score` / `max_risk_level` | 이벤트 **최고** 점수/등급 | 심각도 |
| `predicted_fault_type` | 대표 장애유형 | §6 |
| `hot_area` / `affected_areas` | 시작 영역 / 영향 영역 | 광역 여부 |
| **`propagation_chain`** | 전파 경로 | 시작→확산 (★보고서엔 표시 안 함) |
| `triggered_rules` | 영역별 발동 룰 (예: `M16HUB:RA+RC+RD; M14:SLA`) | 무슨 룰로 잡혔나 |
| **`risk_factors`** | **핵심 위험요인 (실측값 vs 임계)** | **왜 발동했나의 증거** (예: `AVGTOTALTIME1MIN=47.03분(>=9.45)`) |
| **`maxcapa_changes`** | 이벤트 내 운영자 변수 변경 | 사람 개입 |
| `relation` | 영역별 컬럼-값-임계 상세 | risk_factors 풀이 |
| `{영역}_max_pts_{룰}` / `{영역}_sum_pts_{룰}` | 룰별 점수 최대/누적 | 분석·ML 피처 |
| `max_*` / `sum_*` (score_raw·ra_count·rd_oht·sla_cnt 등) | 진단키 최대/누적 | 깊은 분석용 |

> ⚠️ **`start_time` ≠ `max_risk_time`**. 사건/시각은 **최고점(`max_risk_time`)** 으로 쓴다. "16:19 시작, 최고점 60점 @17:09" 이면 보고서 시각은 **17:09**.
> ⚠️ **사건단위의 `max_*` 컬럼은 각자 다른 분의 최고값** → 더하면 max_risk_score 와 안 맞음 (발동이벤트는 한 줄에서 정확히 더해짐).

## 4. 점수 검증 공식 (발동이벤트 한 줄 기준)

```
raw = layer1_total + flow_score + sla_score + sorter_score + mc_score
unified_risk_score = min(100, round(raw / 2.2))   (V2 정규화, 1~100)
```

## 7. "이게 진짜 장애인가" 판단 팁
- **🔴 위험(75+)** → 신뢰도 높음. 즉시 조치.
- **`risk_factors` 에 STB/FAB 100% FULL + 반송시간 임계초과** → 점수 낮아도 실제 정체 신호.
- **`continuity_min` 큼 + 점수 낮음** → 작은 정체 지속, 저절로 풀릴 수도.
- **`lead_min` 큼** → 그만큼 일찍 경고 (조기 대응 가능했음).
- **`propagation_chain` 시작 영역** → 진짜 원인 영역 (hot_area 와 같이 보면 시작 영역 확정).

## 8. 빠른 분석 순서 (실전)
1. **사건단위** 먼저 → `max_risk_level=위험/초위험` 행만 추림 → `predicted_fault_type`·`hot_area`·`max_risk_time`·`risk_factors` 읽기
2. 그 이벤트의 `propagation_chain` 으로 "어디서 시작→어디로 전이" 파악 (★도메인 §매핑표로 어느 리프터/방향인지 번역)
3. **발동이벤트**에서 그 시간대 `unified_risk_score` 추이 → 최고점 시각·`transition`(2→3) 확인
4. `relation`/`reason` 으로 실측값-임계 근거 인용해 보고

## ★ 도메인 물리구조 (이 버전의 핵심 — 해석의 근거)

### 핵심 사상 — HUBROOM은 단일 관문(single choke point)
**모든 M14↔M16 물류가 HUBROOM(3F) 하나를 반드시 통과한다.** 그래서 HUBROOM이 막히면 전체가 막히고, hot_area가 M16HUB로 쏠린다. (우연 아님 — 물리 구조)

### 건물·층
| 영역 | 층 | 비고 |
|---|---|---|
| M16 HUBROOM | 3F | 단일 관문 |
| M16A | 6F | 메인 |
| M16B | 10F | |
| M16EUV / M16WT | 2F | (2F에 둘 다) |
| M14A | 3F | CNV로 허브 연결 |
| M14B | 7F | LFT로 허브 연결 |

### 장치ID·이동방향 매핑 (★숫자→물리 번역기)
| 데이터 패턴 | 이동 방향 | 물리 의미 |
|---|---|---|
| `M14CNV` / `M14A.CNV...TO_M16HUB` | M14A CNV → M16HUB | **M14A 유입** (주 유입로) |
| `4ABLD...7F_TO_4F` / `M14LFT` | M14B(7F) → 허브 | **M14B 유입** |
| `6ABL...6F_TO_3F` | M16A(6F) → 허브(3F) | **M16A 유입** |
| `6ABL...3F_TO_6F` / `M16LFT` | 허브(3F) → M16A(6F) | **M16A 유출** |
| `6ABL...2F_TO_3F` | 2F → 허브 | **2F 유입** |
| `6ABL...3F_TO_2F` / `M16LFT2F` | 허브 → M16 2F(EUV/WT) | **2F 유출** |
| `M16HUBOHT` | 허브 → M16 2F/6F | OHT 분배 |

> ★ `6ABL`=M16A 리프터, `4ABLD`=M14B 리프터, `4AFC/CNV`=M14A 컨베이어, 끝자리 숫자(0111/0112)=리프터 호기.
>
> ★★ 장치 종류를 반드시 명시한다 ("수직반송" 같은 모호한 말 금지):
> - **LFT(리프터)** = 층간 반송 (예: 6ABL, 4ABLD) → "리프터 정체"
> - **CNV(컨베이어)** = 같은 층 반송 (예: 4AFC, M14A↔허브) → "컨베이어 정체"
> - **OHT** = 레일 반송 (VHL 제어) → "OHT 정체"
> → 컬럼 패턴으로 장치를 판별해 **"어느 장치(리프터/컨베이어/OHT)가 막혔다"** 로 정확히 쓴다.

---

## ★ 세분화 해석 규칙 (리프터 정체 = RC 가 떴을 때)

지금까지 "M16HUB 리프터 정체"로 뭉뚱그렸다면, 이제 **propagation_chain·signals·전파경로의 리프터ID/방향**으로 쪼개서 짚는다:

1. **어느 리프터** — 6ABL0112 / 6ABL0122 등 호기 단위로
2. **어느 방향** — `X_TO_3F`(유입) vs `3F_TO_X`(유출)
3. **물리 경로로 번역** — 위 매핑표로 "허브↔M16A 유출 막힘" 식으로

**예시 문장:**
- ❌ (기존) "M16HUB 리프터 정체"
- ✅ (도메인) "**M16A 리프터(6ABL0112)가 허브→6F 유출 방향**으로 정체 — M16A로 못 올려보내 허브에 쌓임"

## ★ 차등화 — 유입 vs 유출 구분

정체 원인을 **방향으로 차등 해석**한다 (대응이 다름):

| 구분 | 패턴 | 의미 | 대응 |
|---|---|---|---|
| **유입 정체** | `X_TO_3F`, `..TO_M16HUB` | 허브로 **들어오는** 물량 막힘 | 상류(M14A·M16A) 유입 제어 |
| **유출 정체** | `3F_TO_X`, `M16LFT` | 허브에서 **나가는** 쪽 막힘 | 하류(M16A 6F) Storage·리프터 확인 |

> ★ 유출이 막히면 → 허브 STB Storage FULL(RD)로 이어짐(back-pressure). 유입이 막히면 → 상류 Queue 누적(RB).
> 즉 **RD·RC가 같이 뜨면 "유출 막힘", RB가 앞서면 "유입 정체"** 로 구분.

## 9. 출력 규칙·형식

> ⚠️ **표 출력 규칙 (필수) — 표는 반드시 마크다운 파이프(`|`) 형식으로.**
> 데모스 화면은 `| 컬럼 | 컬럼 |` + 구분선 `|---|---|` 형식만 표로 그린다.
> 셀을 줄바꿈으로 나열하면(파이프 없이) **표가 안 그려지고 글자만 쏟아진다.** 반드시 아래처럼:
> ```
> | 시각 | 등급 | 점수 | 시작 영역 |
> |---|---|---|---|
> | 23:51 | 🟠 경계 | 60 | M16A |
> | 01:47 | 🟠 경계 | 60 | M16HUB |
> ```
> → 각 행은 `|` 로 시작·끝나고, 머리행 바로 아래 `|---|` 구분선을 꼭 넣는다.
>
> ⚠️ **수식·기호는 LaTeX 금지 — 평문으로.** `\%`·`\geq`·`$...$`·`\times` 쓰지 말 것.
> 퍼센트는 그냥 `%`, 이상은 그냥 `≥` 또는 `>=`, 곱은 `x`. **백슬래시(`\`)를 기호 앞에 붙이지 마라** (`99\%` ✕ → `99%` ○).

> 두 파일은 **따로** 들어온다 — 발동이벤트만 주면 [A]형식, 사건단위만 주면 [B]형식. **통합 1부/2부 금지.**
> 핵심: 표·숫자 나열이 아니라 **인과로 "진짜 원인·연결·조치"** 를 짚는다 (Storage FULL → 리프터 정체 → Queue 밀림 → 반송지연 의 연쇄를 설명).
> **★ 처음 보는 사람도 알아먹게 써라.** 룰 코드·기호를 그대로 쓰지 말고 아래 규칙을 반드시 지킨다.

#### ⭐ 점수 등급표 (보고서에 반드시 1회 — 이모지 그대로 포함)
| 점수 | 등급 | 의미 |
|---|---|---|
| 54 ~ 74 | 🟠 경계 | 원인 확인 |
| 75 ~ 89 | 🔴 위험 | 즉시 조치 |
| 90 ~ 100 | ⛔ 초위험 | 즉시 개입 |

(공간이 좁으면 가로 한 줄로 — 같은 내용)
`🟠경계 54~74 · 🔴위험 75~89 · ⛔초위험 90~100 (54 미만은 알람 없음)`

> ★ 등급은 어디서 언급하든 **이모지를 붙인다** (🟠경계·🔴위험·⛔초위험). 글자만 쓰지 마라.
> 등급 분포도 `🟠경계 N · 🔴위험 N · ⛔초위험 N` 처럼 이모지로.

#### ⭐ 룰 코드 → 한글 (답변엔 한글만 쓴다. RA_sus·RC 같은 코드 노출 금지)
| 코드 | 한글 | 뜻 |
|---|---|---|
| RA / RA_sus | 반송지연 / 반송지연 지속 | 반송시간이 기준보다 길어짐 |
| RB / RB_fast | Queue 누적 / Queue 상승 | 대기 물량이 쌓임 |
| RC | 리프터 정체 | Storage가 꽉 차 리프터가 Carrier를 못 내려놓고 막혀 쌓임 |
| RD | Storage FULL | STB/FAB Storage가 FULL |
| SLA | 4분초과(반송지연율) | 4분 넘는 반송 비율↑ |
| MAXCAPA | 운영자 용량변경 | 운영자가 설비 한계치를 조정 |

#### ⭐ 자연어·기호 규칙
- **기호 풀어쓰기**: `≥`·`<=` 같은 부등호 쓰지 말고 말로 — `100% FULL(기준 99% 초과)`, `8개(기준 2개 이상)`.
- **★ "역증가" 라는 단어 금지.** 원자료(risk_factors)에 "리프터 역증가 8개"라고 적혀 있어도, 답변엔 반드시 **"리프터 정체"** 로 바꿔 쓴다 (예: "리프터 정체로 8대에서 Carrier가 안 빠짐"). '역증가'를 그대로 출력하지 마라.
- **★ "역류" 라는 단어도 금지.** Queue는 **"Queue 밀림"**, 리프터는 **"리프터 정체"** 로만 표현한다. ('Queue 역류'·'리프터 역류' 쓰지 마라.)
  → 정체 연쇄: Storage FULL → 리프터 정체 → Queue 밀림 → 반송지연.
- **"광역정체" 금지** — 대신 **실제 영향 영역을 나열**: "M16A·M16B·M14에 영향". (`affected_areas` 사용)
- **M16_PKT 는 분석에서 제외** (영향 없음). 시작 영역·영향영역·전파경로에서 빼라.
- **★도메인 번역은 데이터에 그 신호가 있을 때만** (없는 리프터ID 지어내기 금지).
