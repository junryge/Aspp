1. 에이전트가 SQL을 "쓰는" 경우 (Text-to-SQL / DB 조회)
LangGraph는 오케스트레이션 프레임워크라서 SQL 실행은 노드나 툴로 그냥 붙이면 된다. 방법은 대략:

노드에서 직접 실행 — 노드 함수 안에서 cursor.execute(query) 돌리고 결과를 state에 넣는다. 제일 단순하고 통제 잘 됨.
LangChain SQL 툴킷 활용 — SQLDatabase, SQLDatabaseToolkit, create_sql_agent. LLM이 스키마 보고 쿼리 생성 → 실행 → 결과 해석하는 흐름. 폐쇄망에서도 SQLAlchemy 드라이버만 있으면 Oracle/MySQL 다 붙는다.
Text-to-SQL 커스텀 그래프 — generate_query → validate → execute → (에러 나면) self-correction 루프. 이게 존이 이미 하던 LangGraph Self-Correction 패턴이랑 딱 맞는다. SQL 문법 에러나 빈 결과를 verifier가 잡아서 다시 생성시키는 식.

2. LangGraph "자체"가 SQL을 쓰는 경우 (체크포인트/메모리)
LangGraph의 상태 영속화에 SQL 백엔드를 쓴다. SqliteSaver, PostgresSaver 같은 체크포인터로 그래프 실행 상태를 DB에 저장. 세션 이어가기나 장기 메모리에 쓴다.
