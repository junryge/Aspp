01 개요 — 뭐냐

한 문장으로: 코딩에 올인한 오픈소스 모델

DeepReinforce가 2026년 6월 공개한, agentic coding(코딩 에이전트) 하나에 특화된 모델 패밀리다. 만능형이 아니라 코딩에 몰빵한 모델이다 — "코드 짜고·고치고·툴 돌리는" 작업에 맞춰 만들어졌다.

// 정체성

코딩 에이전트 특화

자랑하는 지표가 전부 코딩 벤치(Terminal-Bench, SWE-Bench, ClawEval 등). 리포지토리 단위 작업과 툴 사용에 강하다.

// 라인업

4개 사이즈

엣지용 9B Dense부터 프론티어급 397B MoE까지. 31B Dense, 35B MoE 포함. 하드웨어에 맞춰 고른다.

// 개방성

완전 오픈 · MIT

전 체크포인트가 MIT 라이선스로 Hugging Face 공개. 지역 제한 없음. FP8·GGUF 빌드도 함께 배포.

왜 개인 개발에 좋나

MIT + 오픈 가중치 + GGUF 배포 조합이 핵심이다. 클라우드 API에 코드를 안 보내고, 구독료 없이 내 PC(RTX 5090 ×8)에서 그대로 돌릴 수 있다. 라이선스도 자유라 개인 코딩 도구로 쓰기에 문턱이 없다. (개인 PC RTX 3090 24GB 정도여도 35B Q4(21GB)는 돌아간다.)

02 핵심 기술 — Self-Improving

스스로 '풀이 틀'을 짠다

대부분의 코딩 에이전트는 사람이 짜준 고정 harness(scaffold) — 문제 푸는 틀 — 에 모델을 끼워 쓴다. Ornith는 RL 학습 중에 그 틀까지 스스로 생성·최적화하도록 배웠다.[1]

기존 방식

사람이 틀을 짠다

  • 엔지니어가 task별 harness/프롬프트를 수작업으로 설계
  • 모델은 그 고정된 틀 안에서 풀이(solution)만 생성
  • 틀이 한계면 성능도 거기서 막힘
Ornith 방식

모델이 틀도 같이 진화

  • scaffold를 "학습 가능한 객체"로 취급
  • 풀이와 틀을 RL로 동시에 최적화(co-evolve)
  • "어떻게 풀지" + "어떤 틀로 풀지"를 스스로 탐색
오해 주의 — 런타임 자가진화가 아니다

"self-improving"은 학습 방법론 얘기지, 배포 후 알아서 똑똑해진다는 게 아니다. 우리가 받아서 돌리는 가중치는 고정(frozen)이다.

비유하면: 운전 연수 때 스스로 연습 코스를 짜가며 실력을 키운 거지, 면허 딴 뒤 차가 알아서 진화하는 건 아니다. 대화한다고 모델이 학습하거나 기억을 쌓지 않는다.

04 변형 — 어느 걸 쓰나

집·일반 PC는 35B가 천장 — 397B는 데이터센터급

집/일반 PC에서 현실적인 건 9B와 35B다. 24GB급 GPU 한 장(예: 3090·4090)이면 35B Q4(21GB)가 돌아간다. 35B는 MoE라 덩치 대비 추론이 빠른 게 핵심(~3B 활성). 397B는 8장·데이터센터급이 필요해 집에선 못 돌린다 — 일반 사용자는 35B가 천장이다.[1][5]

05 로컬 배포 — 어떻게 돌리나

기존 스택에 거의 그대로 붙는다

표준 추론·에이전트 도구를 다 지원한다. 쓰던 OpenAI 호환 클라이언트/에디터를 그대로 재활용할 수 있다. 집에서 혼자 쓰는 거면 Ollama가 제일 쉬운 길이다.

서빙 / 통합

지원 스택

  • Ollama (가장 쉬움) — ornith:9b (5.6GB) / ornith:35b (21GB) 바로 실행[4]
  • vLLM · SGLang — 멀티 GPU 서빙(397B 같은 대형은 서버급)[2]
  • Transformers (≥ 5.8.1) 직접 로드[2]
  • OpenAI 호환 엔드포인트 (/v1) — 에디터/에이전트 코드 수정 없이 연결[3]
  • 양자화: FP8 · GGUF 공식 빌드 제공[5]
동작 특성

추론 + 툴 호출

  • reasoning 모델 — 답 앞에 <think> 추론 블록 생성[2]
  • 추론 trace는 reasoning_content 필드로 분리 수신 가능[2]
  • well-formed tool_calls 출력 → 에이전트 루프 그대로 구동[3]
  • 컨텍스트 256K (로컬 빌드 기준)[4]
권장 사용처 — 집에서 혼자 개발할 때

전사 배포용으로 끌고 갈 모델이 아니다. 집에서 혼자 개발할 때 개인 코딩 보조로 쓰는 게 제일 잘 맞는다.

① ollama run ornith:9b로 바로 찍먹 → ② 일상은 35B를 Ollama로 띄워 내 에디터/CLI에 붙이기. 일반 PC는 여기까지가 천장이다 — 397B는 8GPU·데이터센터급이라 집에선 못 돌린다. 구독료·외부 API 없이 내 PC에서 끝난다.

06 한계 — 솔직하게

만능형이 아니라, 코딩에 몰빵한 모델이다

장점은 전부 "코딩/에이전트 한정"이다. 도입 전에 기대치를 분명히 해야 한다.

한글
한글 이해도는 많이 떨어진다 — '알아먹는' 정도

코딩 데이터로만 튜닝돼 한국어 자연어 능력은 강점이 아니다. 베이스(Qwen 3.5/Gemma 4) 덕에 한글 지시를 대충 알아먹고 코드로 옮기는 정도는 되지만, 뉘앙스 파악·긴 한글 독해·한글 작문/번역은 확실히 약하다. 한글 지시→코드는 그럭저럭, 한글 문서·설명·번역은 별도 모델(Qwen3.6/GLM/Gemma) 권장.

범용
코딩 외 일반 지식 Q&A는 범위 밖

요약·잡담·일반 상식 같은 범용 작업은 general 모델이 낫다. 역할을 코딩 엔진으로 한정해야 효과가 산다.

체급
"Opus급"은 397B 얘기다

화제의 벤치 수치는 397B 기준. 우리가 돌릴 9B·35B는 그만큼 안 나온다(9B TB 43.1, 35B TB 64.4).[1] 기대치를 변형별로 맞춰야 한다.

상한
전 세계 1등은 아니다

397B도 Opus 4.8·GLM-5.2-744B에는 밀린다.[5] "오픈모델 동급 최강, 집에선 35B까지"가 정확한 포지션.

07 제언 — 결론

집에서 혼자 코딩 개발 용도로 써라

한 줄 요약

코딩 전용 모델이다. MoE 효율 · 스스로 scaffold 생성 · 깔끔한 tool calling · MIT 오픈.
만능 챗봇 아니다 — 한글 문서·번역·범용은 다른 모델(Qwen3.6/GLM/Gemma) 써라. 구독료·외부 API 없이 집 PC에서 혼자 코딩 보조로 굴리기에 현재 가장 현실적인 오픈 선택지 중 하나.

1찍먹: 9B(5.6GB)를 ollama run ornith:9b로 깔아 코드 품질·추론 trade-off 확인. 5분이면 감 잡힌다.
2일상화: 35B를 Ollama로 띄워 에디터/CLI에 개인 코딩 어시로 연결. 일반 PC는 35B가 천장 — 397B는 8GPU·데이터센터급이라 집에선 못 돌린다.
3분리 운용: 코딩은 Ornith, 한글 문서·범용은 Qwen3.6/GLM/Gemma로 — 용도별 모델 조합.
